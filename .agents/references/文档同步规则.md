# 文档同步规则

> 按需加载：当你要写代码并需要同步文档时，读取本文件。

---

## 核心原则

文档更新是代码变更的原子步骤，不是事后补充。每个代码变更必须同步完成文档更新，否则该变更不算完成。

## 代码变更前的文档影响评估

**在开始写代码之前**，必须先回答：本次变更会影响哪些文档？

```
代码变更 → 文档影响评估 → 确认受影响文档列表 → 开始实现 → 同步更新文档 → 验证一致
```

具体做法：在 todo list 中，每个代码任务后面标注受影响的文档。例如：

```
- [ ] 新增 /api/sources 路由 → 影响：API接口.md, 技术架构.md(项目结构)
- [ ] Prisma 新增 Source model 字段 → 影响：技术架构.md(数据模型), API接口.md(响应)
```

**更新流程**：识别到图表相关变更 → 打开 Excalidraw 在线链接 → 编辑图表（fontFamily: 2，中文标注） → 导出新链接 → 更新文档中的链接。

**图表风格**：字体 fontFamily: 2（Helvetica）；中文标注组件名和职责，英文标注技术术语；颜色按层级区分（蓝色=用户层、紫色=API层、绿色=Agent层、青色=数据层、橙色=沙箱层）。

## 文档更新检查清单

每个独立任务完成时：

```
□ 代码变更已完成
□ 受影响的文档已全部更新
□ 文档中的代码示例/Schema 与实际代码一致（非旧版本）
□ 文档版本号和更新日期已刷新
```

> 如果代码变更已完成但文档未更新，该任务不得标记为 completed。

## 文档自检方法

| 文档                   | 验证方法                                                      |
| ---------------------- | ------------------------------------------------------------- |
| `API接口.md`           | 接口列表 ↔ `apps/server/src/routes/` 目录下的路由文件数量一致 |
| `技术架构.md` 数据模型 | Prisma Schema ↔ 文档中的 prisma 代码块完全一致                |
| `技术架构.md` 环境变量 | `.env.example` ↔ 文档中的变量列表完全一致                     |
| `技术架构.md` Docker   | `docker-compose.yml` ↔ 文档中的 yaml 代码块一致               |
| `Prompt设计.md` 工具   | `apps/worker/src/agent/tools/*.ts` ↔ 文档中的工具定义一致     |
| `迭代计划.md`          | 已完成的迭代 ↔ `git log` 中的实际提交                         |

## 代码变更登记规则

代码变更后如果来不及同步文档，必须登记到开发日志的「待同步文档变更」区域。

**格式**：

```markdown
- [YYYY-MM-DD HH:MM] `文件路径` | 变更描述 | 影响: 文档1(章节), 文档2(章节)
```

**触发条件**：新增/修改 API 路由、修改 Prisma Schema、新增 Agent 工具、修改 Prompt、新增技术决策等。

**不需要登记的**：UI 样式调整、注释修改、测试文件修改、代码格式化。

**消费流程**：每个 session 启动时检查开发日志的「待同步文档变更」区域，有条目则按影响的文档分组更新，完成后删除条目。
