# 质量门控

> 按需加载：当你要提交代码或跑测试时，读取本文件。

---

## 构建检查（每次 commit 前必须通过）

```bash
pnpm --filter @ai-teacher/web build
```

## 单元测试（每次 commit 前必须通过）

```bash
pnpm test:unit
```

> 单元测试覆盖核心逻辑（加解密、掌握度门控、解析器、Prompt 组装等），commit 前必须通过。覆盖率检查用 `pnpm test:coverage`。

## 文档一致性检查（每次 commit 前必须通过）

```bash
pnpm check:docs
```

> 检查 Prisma Schema、环境变量、API 路由、Agent 工具、端口表与对应文档的一致性。文档与代码不一致时必失败，按 AGENTS.md §3.3 修正。
>
> ⚠️ `.agents/references/` 的时效性不在 check:docs 自动覆盖范围内（规则性文档无法 diff）。涉及构建工具/技术栈/端口配置变更时，手动检查 `缓存与重启.md`、`质量门控.md` 等 references 是否需同步（见 `文档同步规则.md` 触发映射表）。

## Lint 检查（每次 commit 前必须通过）

```bash
pnpm lint
```

> ESLint flat config（根 `eslint.config.mjs`）+ 自定义规则 `no-default-export`（`scripts/eslint-rules/`），强制 AGENTS.md §4 命名导出规范。错误信息含 AI 可消费的修复提示。config 文件（eslint/vite/postcss）与 Playwright globalSetup/Teardown 因框架要求 default export 而豁免。
>
> `@typescript-eslint/no-explicit-any` / `consistent-type-imports` / `no-unused-vars` 为告警级（非阻塞），属既有技术债，不在迭代 042 范围。

## 依赖方向检查（每次 commit 前必须通过）

```bash
pnpm check:deps
```

> 自定义脚本 `scripts/check-deps.mjs`（ArchUnit 风格），强制 AGENTS.md §5 模块依赖方向：apps/web→shared、apps/server|worker→shared+db、packages 为叶子包。采用自定义脚本而非 dependency-cruiser（后者要求 Node ^22||^24||>=26，与 CI Node 20 / 本地 Node 25 不兼容）。
>
> 已知技术债（5 处 server↔worker 共享代码逃逸）以白名单显式豁免，见 `决策记录.md` ADR-024，待后续迭代「依赖方向治理」归位后移除。新增违规会失败。

## CI 流水线（push/PR 自动运行）

GitHub Actions（`.github/workflows/ci.yml`）在 push/PR 时自动运行 `typecheck` + `lint` + `check:deps` + `test:unit` + `check:docs` + `web build`。

## E2E 测试（发版时运行，日常不强制）

```bash
pnpm test:e2e
```

> E2E 测试需要启动完整的 Web + Server + Worker + 数据库环境，日常频繁提交时跑 E2E 会严重影响开发效率。

## E2E 测试同步规则

当需求变更涉及 UI 行为、页面路由、API 接口变化时，**必须同步修改对应的 E2E 测试用例**。

先查迭代 ↔ E2E 映射表（在 `docs/开发/迭代计划.md` 的"迭代 ↔ E2E 测试映射"章节）。

| 变更类型                       | 必须检查/修改的 E2E 文件                        |
| ------------------------------ | ----------------------------------------------- |
| 修改页面路由或重定向行为       | `e2e/home.spec.ts` 及涉及路由的测试             |
| 修改 Chat UI 交互              | `e2e/chat.spec.ts`, `e2e/learn.spec.ts`         |
| 修改诊断流程                   | `e2e/diagnostic.spec.ts`                        |
| 修改会话管理（新建/归档/状态） | `e2e/session-management.spec.ts`                |
| 修改掌握度/节点推进逻辑        | `e2e/mastery.spec.ts`                           |
| 修改快问/建议回复功能          | `e2e/quick-features.spec.ts`                    |
| 新增 API 路由                  | 新增对应 `e2e/*.spec.ts` 或在已有文件中追加用例 |
| 修改推荐话题/首页内容          | `e2e/home.spec.ts`                              |

## E2E 失败处理

1. 分析失败原因：代码 bug 还是测试用例过时？
2. 代码 bug → 修复代码 → 重新跑 E2E → 通过后提交
3. 测试过时 → 确认新行为符合预期 → 更新测试 → 重新跑 E2E → 通过后提交
4. 绝不允许：跳过测试、删除失败测试、@ts-ignore 测试错误
