# 实施纪律（SDD 节律）

> 按需加载：当你要写代码实现迭代任务时，读取本文件。
>
> 把单个迭代任务的实施环节固化为 spec → plan → TDD → verify 节律，补强 AGENTS.md §3.0 的执行层。以工具无关的纯文档承载，不依赖任何 AI 工具的 skill 机制。

---

## 适用范围

- **适用**：涉及代码变更的迭代任务（产品需求 / 问题修复 / 技术改进）
- **不适用**：纯文档更新、文案调整、CSS 微调等无逻辑验证需求的小改动（见 `大改动讨论.md` 的"小改动直接执行"）

---

## 1. Todo 结构化

每个 todo 项**必须**含三要素：

- **动作**：动词 + 对象（"新增 X 路由"），不写"处理 X 问题"这类模糊表述
- **文件**：受影响路径（多个逗号分隔）
- **验证**：如何确认完成——测试命令 / E2E 文件 / 手动检查步骤

格式：

```
- [ ] {动作} → 文件: {路径} | 验证: {方式}
```

> ⚠️ 没有验证方式的 todo 视为不合格，必须补充后才能开始执行。

## 2. 逐项验证不跳步

- **一次只推进一个 todo**，验证通过后才进入下一项
- 禁止"批量改完所有文件，最后统一验证"——错误定位成本会爆炸
- 验证失败 → 立即修复当前项 → 重新验证 → 通过后再进下一项
- 不允许跳过失败的项继续往下做

## 3. TDD 优先（适用时）

涉及**可测试逻辑**时，测试先行：写失败测试（红）→ 实现最小逻辑（绿）→ 必要时重构。

**强制 TDD**（可测试逻辑）：
- 纯函数 / 数据转换（评分计算、token 估算、加解密）
- 解析器（block-parser、流式增量解析）
- 业务规则（掌握度门控、节点推进、状态机转换）
- Zod Schema 边界校验

**不强制 TDD**：
- UI 组件 / 样式（用 E2E 覆盖）
- Prompt 文本（用人工 + E2E 评估）
- 配置 / 脚手架代码

> 基础设施依赖：TDD 强制化以迭代 045（Vitest 单元测试基础设施）完成为前提。045 完成前，对已有测试覆盖的模块优先 TDD；045 完成后，上述"可测试逻辑"全面强制。

## 4. Spec 同步（方案偏离先改文档）

实现过程中如果发现迭代详情方案不可行 / 需要偏离 / 有遗漏：

```
发现偏离 → 暂停实现 → 更新迭代详情文档（方案/步骤/验收标准） → 继续实现
```

> 禁止"在心里改方案"——方案变了但文档没变，后续 session 会基于旧方案做过时性判定，造成混乱。本条是 AGENTS.md §3.3（文档时效性）在实施层的具体化：§3.3 管"文档与代码不一致"，本条管"方案与实现偏离"。

---

## 收尾验收

所有 todo 完成后，**逐条对照迭代详情的"验收标准"checkbox 勾选**：

- 每条验收标准都有对应已完成且验证通过的 todo
- 全部勾选 → 迭代视为完成
- 任一无法勾选 → 不能标记迭代完成，补做对应工作

完成后按 `文档同步规则.md` + `质量门控.md` 收尾（文档同步 + E2E 检查）。

---

## 与其他规则的关系

| 规则 | 职责 | 与本文件关系 |
|------|------|-------------|
| `AGENTS.md` §3.0 | 方案 review + todo 规划前置 | 本文件接在 §3.0 之后，规范 todo 的**结构和执行** |
| `文档同步规则.md` | 文档影响评估 + 触发映射 | 本文件 §1 要求 todo 标注验证方式；文档同步由该文件管 |
| `质量门控.md` | 构建 / E2E 检查 | 本文件 §2 的"验证"常引用此处的测试命令 |
| `大改动讨论.md` | 大改动先讨论 | 讨论通过后才进入本文件的实施节律 |
