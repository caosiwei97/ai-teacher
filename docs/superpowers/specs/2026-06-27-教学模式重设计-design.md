## 0. 背景：为什么要重设计

### 当前痛点（根因，已用代码证据确认）

当前教学模式"啰嗦"是**结构性**问题，来自四个叠加因素：

| 根因           | 代码证据                                                                                       | 后果                                        |
| -------------- | ---------------------------------------------------------------------------------------------- | ------------------------------------------- |
| 纯苏格拉底串行 | `apps/worker/src/agent/prompts/tutor.ts:64`（规则1"先铺垫+追问"）、`:68`（规则5"2-3轮才总结"） | 单知识点要 2-4 轮对话才能过，每轮带铺垫文本 |
| 掌握仪式       | `apps/worker/src/agent/tools/assess-mastery.ts:81-85` 硬编码四步（报告+庆祝+桥接+停）          | 每过一节点强塞一大段总结                    |
| 诊断仪式       | `tutor.ts:86-98` 打招呼→出题→分析→路线图→过渡                                                  | 开场跨多轮                                  |
| 表现力单一     | `apps/web/src/components/ui-blocks/registry.tsx` 仅 12 种固定卡片，**无 iframe/HTML**          | 概念只能靠"反复追问"确认理解，越追问越啰嗦  |

**关键洞察**：第 4 条是前 3 条的放大器。因为没有任何可视化/互动手段让用户**自己看懂**，agent 只能靠**多轮追问**确认用户懂没懂。

### 重设计核心主张

> **从「对话主导、卡片点缀」翻转为「互动教学产物主导、对话辅助」。**
>
> 把"本来要靠 4-6 轮追问才能讲清的概念"，变成一份用户自己看+练的互动课。对话退化为「答疑 + 追问 + 判定掌握」的轻角色。

### 三个教学理念的收敛

| 维度       | 当前 ai-teacher       | teach (Pocock)                                   | sigma (2-Sigma)      |
| ---------- | --------------------- | ------------------------------------------------ | -------------------- |
| 核心产物   | 每轮对话（铺垫+追问） | **一节自包含 HTML 课**——短、漂亮、一次一个小胜利 | 每轮 1-2 个问题      |
| 知识怎么来 | agent 即兴生成        | **先从可信资源聚合**，课里只讲"够用的"           | 直接提问、不讲解     |
| 技能怎么练 | 追问确认              | **交互式 HTML 练习 + 紧密反馈环**                | 掌握检查 + 实践任务  |
| 记忆强度   | 无设计                | 流利度 vs 储存强度（desirable difficulty）       | 间隔重复 + 交错      |
| 可视化     | 几乎没有              | **课=HTML**（动画/交互/思维导图全在课里）        | 大量 HTML/Excalidraw |

**收敛结论**：对抗"啰嗦"最锋利的武器是 teach 的"课 = 自包含 HTML 互动产物"理念。

## 1. 产品愿景

### 三阶段闭环：🌱 学 → 🔁 固 → 🔥 验

```
🌱 学习模式          →    🔁 复习模式        →    🔥 面试模式
（掌握新知识）            （对抗遗忘）            （高压检验）
互动课 + 教练式追问       间隔重复 + 提取练习     面试官拷问 + 评分复盘
```

| 模式    | Agent 角色          | 来源范围                 | 主要产物               | 结束输出                     |
| ------- | ------------------- | ------------------------ | ---------------------- | ---------------------------- |
| 🌱 学习 | 私教（引导+鼓励）   | 单个知识点，按路线图推进 | 互动课/动画/思维导图   | 掌握报告 + 推进路线图        |
| 🔁 复习 | 考官（提问+核对）   | 单章/跨章节串联          | 抽认卡/回忆测验/错题本 | 记忆强度 + 下次复习时间      |
| 🔥 面试 | 面试官（拷问+施压） | 单章/全局/指定主题       | 面试题/评分卡/复盘     | 面试评分 + 薄弱点 + 改进建议 |

### 重要澄清：现有 `interviewer` ≠ 面试模式

当前 `tutor.ts:36` 的 `interviewer` 是"学习过程中的追问风格"，本质还是在教。本次设计的**面试模式是独立模式**：不教新东西，拿已学内容来检验——出题、追问、施压、评分复盘。两者不共用。

## 2. 学习模式

### 2.1 互动产物形态（A 为标准，BCD 按需嵌入）

**形态 A：自包含互动课（teach skill 理念）— 标准形态**

一节自包含 HTML 卡片，三段式结构：

1. **概念**（1 句话）—— 最小上下文
2. **动手感受** —— 可交互区域，用户自己操作感受
3. **自测**（1 题）—— 即时检验

**形态 B：单步动画演示（teaching-viz 理念）— 辅助**

步进式动画，用户控制播放，展示"看不见的过程"。嵌入 A 的"动手感受"区。

**形态 C：互动思维导图（sigma 理念）— 辅助**

展示知识点间的结构关系，点节点跳转/展开。用于章节总览/串讲。

**形态 D：可运行代码沙箱 — 复用现有**

复用现有 OpenSandbox（`apps/worker/src/agent/tools/execute-code.ts`）。代码类知识点嵌入 A 的"动手感受"区。

**分级关系**：A 是骨架，BCD 是 A 的肌肉。Agent 根据知识点性质自动选择"动手感受"区的内容。

### 2.2 对话角色：教练式

**核心原则：对话是"辅助轮"，不是"发动机"。**

| 时机       | 行为                                                            |
| ---------- | --------------------------------------------------------------- |
| 发产物时   | 产物 + 一句话引导（"自己看+练，有疑问随时问"）                  |
| 用户提问时 | 答核心 + 反问深入（保持苏格拉底但更短）                         |
| 用户沉默时 | **绝不主动开口**（不铺垫、不复述、不催促）                      |
| 判定掌握后 | 一句确认 + 推进（"✓ 掌握，下一节：X"），**删掉现状的 4 步仪式** |
| 用户答错时 | 构造对比场景引导，让用户自己发现                                |

**对话量目标**：每知识点从现状的 4-6 轮 → 1-2 轮。

### 2.3 Prompt 改造点

- 放宽 `tutor.ts:64`（规则1 铺垫）：产物已承担讲解，对话不再铺垫
- 删除 `tutor.ts:69`（规则5 2-3轮总结）的强制轮数
- 精简 `assess-mastery.ts:81-85` 的四步仪式，合并为一句

## 3. 复习模式

### 3.1 范围：智能推荐 + 主动指定

**智能推荐（默认）**：基于间隔重复算法自动算出"今天该复习什么"。基于每个知识点的**记忆强度 + 上次复习时间 + 掌握度**，到期才出现。跨章节混合（交错复习，记忆更牢）。

**主动指定（补充）**：用户手动圈定单章/多章串联/指定知识点。适合考前突击、系统性回顾。

### 3.2 产物：提取练习三件套

**核心区别**：复习**不重新讲概念**（那是学习模式的事），而是**让用户主动回忆**——答对放行，答错才提示。

- 🎴 **抽认卡 Flashcard**：正面问题 → 翻面答案
- ✍️ **回忆测验 Recall**：自由作答 → AI 评分查漏
- 🔍 **错题本 + 思维导图**：复习结束自动汇总薄弱点

### 3.3 结束输出

不做现状的"4 步仪式"，只输出：

- 记忆强度评分（强化/维持/衰退）
- 下次复习时间（基于算法自动排期）
- 薄弱点清单（可一键转回学习模式重学）

### 3.4 间隔重复算法

复用 sigma 的间隔策略（1d→2d→4d→8d→16d→32d），但持久化到 DB（每个知识点的记忆强度、下次复习时间）。

## 4. 面试模式

### 4.1 四阶段流程

```
🎯 设定 → 🔥 拷问（核心）→ 📝 每题即时评分 → 📊 复盘
```

1. **设定**：用户选范围+难度，面试官读"开场白"，进入角色
2. **拷问**：出题 → 听答 → 追问。答得好追问更深，答不好追问"为什么"。**绝不讲解**
3. **每题即时评分**：每题答完即时评分，标注薄弱点，决定下一题难度
4. **复盘**：总评分卡 + 薄弱点清单 + 改进建议

### 4.2 三档难度 + 动态调整

| 难度    | 题型               | 追问深度              | 通过线            |
| ------- | ------------------ | --------------------- | ----------------- |
| 🟢 初级 | 概念辨析、定义解释 | 追问 1 层"为什么"     | 答对核心即可      |
| 🟡 中级 | 代码预测、调试改错 | 追问 2-3 层，要求权衡 | 答对 + 能解释取舍 |
| 🔴 高级 | 系统设计、开放问题 | 持续追问，挑战假设    | 答对 + 能反驳反例 |

**动态调整**：连续答对 → 升难度；连续答错 → 降难度但记录为薄弱点。始终保持"刚好够呛"的强度。

### 4.3 与复习模式的关键区别

| 维度     | 🔁 复习                | 🔥 面试                      |
| -------- | ---------------------- | ---------------------------- |
| 提示策略 | 答错才提示             | **绝不提示**，只追问         |
| 答错处理 | 标记到错题本，放行     | 追问"为什么错"，深挖到底     |
| 压力基调 | 温和（保学习积极性）   | 高压（模拟真实面试）         |
| 产物形态 | 抽认卡/回忆测验/错题本 | 纯对话+代码块/评分卡/复盘    |
| 是否讲解 | 答错后简短提示         | **全程不讲解**，复盘才给建议 |

### 4.4 语音能力：混合一步到位

**策略**：流式 TTS + 按钮式 STT。

- **TTS（面试官说）**：流式合成（DashScope CosyVoice），LLM 流式输出 → 文本流实时喂给 TTS → 边生成边播放，面试官开口快
- **STT（用户说）**：按钮式（按住说话，松开提交），规避端点检测和打断两大难点
- **供应商**：复用现有 DashScope 体系
- **升级路径**：后续按钮式 STT → 全双工实时 STT。**注**：全双工涉及 VAD（端点检测）/打断/双向音频流，前端接口**尽量保持兼容但不可能完全不变**，升级时需重做实时音频流组件。

**为什么不做全双工**：TTS 容易，STT 是真难点（端点检测、麦克风权限、环境噪声、打断逻辑）。这些是"装上去用了才知道行不行"的体验问题，需要真人反复调。混合一步到位把能一次做对的（流式 TTS）做到终态，把需人肉调优的（全双工 STT）留接口、先用简化版。

## 5. UI 设计

### 5.1 布局：统一工作台

**三模式共享同一个三栏布局**，差异只体现在中栏顶部 Tab + 对话内容 + 右栏。模式切换不是跳转新页面，而是**切换状态**——保持认知连续性。

```
┌─────────┬──────────────────────────────┬──────────┐
│         │  [🌱学习] [🔁复习] [🔥面试]   │          │
│ 会话列表 ├──────────────────────────────┤  右栏     │
│ (左栏)  │                              │ (随模式  │
│         │      对话区 + 互动产物         │  变化)   │
│         │                              │          │
└─────────┴──────────────────────────────┴──────────┘
```

**右栏随模式变化**：

- 学习 → 路线图（节点列表 + 总进度）
- 复习 → 今日复习清单（到期项 + 完成度）
- 面试 → 面试评分卡（实时分数 + 薄弱点）

### 5.2 渐进解锁

| 状态      | 条件       | Tab 表现                                         |
| --------- | ---------- | ------------------------------------------------ |
| 0 知识点  | 全新会话   | 复习/面试 Tab 灰锁，hover 提示"先学完一个知识点" |
| ≥1 知识点 | 掌握第一个 | 解锁，范围=当前已掌握知识点                      |
| 多知识点  | 持续学习   | 复习可串联，面试可综合                           |
| 全掌握    | 学完整图   | 面试可做"结业模拟"，学习 Tab 可进入"已完成"态    |

**复习/面试范围始终 = 当前已掌握的全部知识点**，随学习推进自动增长。这既满足"学完才能复习/面试"，又抓住间隔重复黄金期（学完一节当天就能复习）。

> **单知识点的复习降级**：只有 1 个已掌握知识点时，"跨章节交错"无意义。此时复习产物降级为**单点提取练习**（对该知识点的抽认卡/回忆测验），不做交错混合。掌握 ≥2 个知识点后才启用交错复习。这不是矛盾——当天复习符合间隔重复黄金期，只是产物形态按知识点数量自适应。

### 5.3 全页面设计

按用户旅程顺序：

**① 首页落地页**（替代当前 `home.tsx` 的纯重定向）

- 大标题"真正掌握，而不只是看过" + 价值主张
- 居中输入框（输入主题即开学）+ 推荐主题 chips
- 底部三阶段闭环示意
- 极简，无多余导航

**② 新会话起步页**（输入主题后、诊断前）

- 把 teach 的 MISSION + sigma 的诊断前置，做成轻量卡片式
- 2 个问题（动机 + 水平），点选即可，**可跳过**
- 替代当前"一进会话就 agent 主动出 3-5 道诊断题"的重仪式

**③ 左侧会话列表**（保留现状分组，重新设计视觉）

- 分组：新对话 / 学习中 / 已完成
- 每个会话前加模式图标（🌱/🔁/🔥）
- 进度条颜色随模式（学习靛蓝/复习黄/面试红）
- "新对话"按钮提权为显眼按钮
- 已完成显示 ✓ 并弱化

**④ 工作台·学习态**：互动课内嵌气泡，右栏路线图
**⑤ 工作台·复习态**：抽认卡内嵌，右栏复习清单（黄色基调）
**⑥ 工作台·面试态**：纯对话+代码，底部麦克风，右栏评分卡（红色基调）

### 5.4 响应式设计（桌面 + 超宽屏）

**范围**：只支持桌面/笔记本 + 超宽屏，手机后置（MVP 不专门优化）。

| 断点   | 宽度                   | 布局                    |
| ------ | ---------------------- | ----------------------- |
| xl/2xl | ≥1440px 超宽屏         | 三栏 + **产物聚焦面板** |
| lg     | 1024-1440px 标准笔记本 | 三栏（默认形态）        |
| <lg    | <1024px                | 降级（不崩坏即可）      |

**超宽屏核心创新：产物聚焦面板**

- 屏幕宽于 1440px 时，对话区裂成两栏
- 左边对话流（限宽 680px 保持可读）
- 右边产物聚焦面板，点对话里互动课的"展开"图标，互动课/动画/思维导图在大屏完整呈现
- 避免超宽屏内嵌气泡要么撑爆宽度、要么居中留空的问题

**笔记本形态**：左栏折叠图标栏 + 互动课内嵌 + 右栏可折叠
**右栏折叠后**：对话专注模式，浮标（"🗺 路线 3/9"）可唤回

**核心理念**：不是"所有屏幕塞下所有东西"，而是"不同屏幕展现不同层次的信息"。但学/固/验三模式 + Tab 切换在所有桌面尺寸一致可用。

**手机明确后置**：不崩坏即可；面试语音桌面专用。

### 5.5 风格统一原则（贯穿所有页面）

1. **配色统一**：全部用 `globals.css` 已有变量。三模式靠主色点缀区分（学习靛蓝/复习黄/面试红），底色始终 #0f1117
2. **布局统一**：所有页面都是左栏+中区+右栏三栏；落地页/起步页是中区特例（右栏隐藏）
3. **产物统一**：互动课/动画/抽认卡/面试题都内嵌对话气泡，不弹窗不跳转
4. **交互统一**：卡片圆角 10px、按钮一致、进度条一致、过渡动效统一（`fadeSlideIn`/`message-in`）
5. **模式切换统一**：顶部 Tab 始终在位（未解锁则灰锁），切换是状态变化非页面跳转
6. **渐进解锁统一**：复习/面试 Tab 灰锁→解锁遵循"≥1 知识点"规则

## 6. 技术实现路径

### 6.1 新增能力：互动产物渲染（HTML 沙箱）

当前无 iframe/HTML 渲染能力。需新增：

1. **Schema**：`packages/shared/src/schemas/ui-block.ts` 新增 block type `interactive`（携带自包含 HTML）
2. **前端渲染器**：`apps/web/src/components/ui-blocks/registry.tsx` 加 iframe 沙箱渲染器（`srcDoc` + `sandbox` 属性，隔离脚本）
3. **后端 tool**：`apps/worker/src/agent/tools/render-ui.ts` 的 `z.discriminatedUnion` 放开对应 type
4. **安全**：iframe 用 `sandbox="allow-scripts"`（不允许 same-origin），HTML 由 agent 生成需做基本净化

> ⚠️ **既有技术债：`render-ui.ts` 与 `ui-block.ts` 已不同步。**
> 现状是 `render-ui.ts` 的 union 只放开 **6 种**（table/callout/comparison/heading/badge/mastery-report），
> 而 `ui-block.ts` schema 有 **12 种**（多出 text/assessment/quiz/code-result/formula/diagram）——后者由其他路径产出，不经过 renderUI tool。
> 二者本就脱节。**新增 `interactive` 时必须两处同步改**，实现计划应借机梳理这个既有债（至少记录为已知问题，明确每个 block type 的产出路径）。

> ⚠️ **产物失败必须有降级路径。** 产物主导模式下，产物挂了等于教学中断。见 §8.1 降级设计。

### 6.2 三模式的 Agent 编排

当前是隐式状态机（Prompt + tool 副作用驱动）。三模式实现：

- **共享**：同一套产物能力（ABCD）、同一套底层（路线图/掌握度/沙箱/RAG）
- **差异**：Agent 人格（system prompt）+ 触发规则 + tool 集合
  - 学习：教练式 prompt + 互动课 tool
  - 复习：考官 prompt + 抽认卡/回忆测验 tool + 间隔重复评分
  - 面试：面试官 prompt + 评分 tool + 语音 TTS/STT

**实现倾向**：遵循迭代 041 的"先内联，成熟再抽象"原则，三模式先各自独立 prompt + tool，跑通后再考虑抽象。

### 6.3 间隔重复持久化

新增数据：每个知识点的记忆强度、上次复习时间、下次复习时间、复习间隔。

### 6.4 语音技术栈

- **TTS**：DashScope CosyVoice 流式合成（`@` SDK 或 HTTP）
- **STT**：DashScope Paraformer（按钮式：录音 → 上传 → 转写）
- **前端**：Web Audio API（播放）+ MediaRecorder（录音）
- **传输**：TTS 音频流走现有 SSE 通道新增 audio chunk 事件；STT 录音文件上传 server 转发 DashScope

### 6.5 响应式实现

用 Tailwind 4 原生断点 `lg:`(1024) / `xl:`(1280) / `2xl:`(1536)，核心逻辑写在 lg 和 xl 上，无需额外响应式库。产物聚焦面板用 `xl:` 条件渲染。

### 8.1 产物失败降级设计

产物主导模式下，**产物挂了等于教学中断**，必须有降级路径：

| 失败场景                                                     | 降级策略                                                                                                                                     |
| ------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------- |
| Agent 生成的互动课 HTML **畸形/超大**（>设定阈值，如 100KB） | 前端 iframe 沙箱加载超时/失败 → 回退展示"产物加载失败"+ 代码原文（用现有 code-block）+ 对话补一句文字讲解                                    |
| Agent 生成的 HTML **含恶意/危险脚本**                        | `sandbox="allow-scripts"`（无 same-origin）已隔离；额外做基本净化（移除 `<script src=外部>`、`javascript:` 协议）。净化失败 → 同上降级到文字 |
| **互动课生成失败**（LLM 未产出 interactive block）           | 对话回退到"纯文字讲解"（教练式 prompt 兜底：产物没出来时，用文字讲清概念+追问）                                                              |
| **iframe 沙箱渲染异常**（浏览器兼容性/内存）                 | React ErrorBoundary 捕获 → 显示 fallback UI（"此处互动内容无法显示，已转为文字"）                                                            |
| TTS/STT **失败**（Phase 0 未配 DashScope Key、网络超时）     | 面试模式自动降级为纯文本面试，提示"语音不可用，已切换文本模式"                                                                               |

**原则**：任何产物失败都不得让教学流程卡死。降级到文字讲解是最后兜底——这也是为什么 §2.2 教练式对话角色保留"答核心+反问"能力，而非完全依赖产物。

### 8.2 三模式 Agent 编排的抽象触发线

§6.2 遵循迭代 041"先内联，成熟再抽象"。但为避免"永远不抽象"或"过早抽象"，预设抽象触发条件：

- 三模式 system prompt 的**公共部分（角色/工具说明/输出规范）重复度 > 60%** → 抽出 `buildBasePrompt()` + 各模式 `modeSection()`
- 三模式共享的 tool（路线图/掌握度/沙箱/RAG）**调用包装重复 > 3 处** → 抽出 `createSharedTools()`
- 触发条件未达到前，保持内联独立，不强行抽象

## 9. 确认的 9 项核心决策

1. **大方向**：产物主导，对话辅助（非对话主导）
2. **学习产物**：形态 A（自包含互动课）为标准，BCD 按需嵌入
3. **学习对话**：教练式 — 产物讲概念，对话只引导/答疑/判定
4. **复习**：智能推荐为主 + 提取练习三件套（不重讲）
5. **面试**：设定→拷问→即时评分→复盘，全程不讲解只追问
6. **语音**：混合一步到位（流式 TTS + 按钮式 STT）
7. **UI**：统一工作台，三模式顶部 Tab 切换
8. **解锁**：渐进解锁，掌握≥1 知识点即开放复习/面试
9. **模型配置**：.env 回填 DB（UI 全管）+ 首次软引导 + 增强能力 KEY 纳入 DB

## 10. 模型配置引导（Phase 0）

### 10.1 现状问题

当前有**两套并行的 KEY 体系**，割裂且无引导：

- **体系 A（.env）**：4 类 KEY 散落在 `.env`（`OPENAI_API_KEY`/`OPENAI_BASE_URL`、`EMBEDDING_API_KEY`、`DASHSCOPE_API_KEY`、`LLM_ENCRYPTION_KEY`）。用户要手动改文件，无 UI 无引导，改错全功能不可用。`getFallbackProvider()`（`packages/shared/src/services/provider-registry.ts:86`）直接读 `process.env.OPENAI_API_KEY`。
- **体系 B（DB）**：settings 页面（`apps/web/src/pages/settings.tsx`）**已有完整 LLM 配置 UI**（添加/删除/测试/设默认，支持 9 个 provider，KEY 加密存 DB）。但页面空着没引导，且**只管 LLM，不管 Embedding/视频 KEY**。

**首次启动用户面对**：一个空白的 settings 页 + 一堆要手填的 .env 变量，没有任何引导告诉他"先配哪个"。

### 10.2 统一方案：.env 回填 DB（UI 全管）

| 决策               | 选择                                                                      |
| ------------------ | ------------------------------------------------------------------------- |
| 统一方向           | 方案 ①：.env 回填 DB，UI 成为所有 KEY 的唯一入口，.env 仅作"首次预填"来源 |
| Embedding/视频 KEY | 纳入 DB 管理（扩展 settings 配置体系），不再只走 .env                     |
| 首次引导           | **软引导**：主 LLM 就绪即可进入，可选功能（视频/RAG）用到时再提示         |

**回填与覆盖优先级**：

1. 启动时检测 `.env` 的 KEY → 自动回填到 DB 配置，标记"来自环境变量"
2. 回填仅当 DB 中无对应配置时进行（不覆盖用户已在 UI 配置的）
3. 运行态统一读 DB 配置；UI 修改后以 DB 为准（DB 覆盖 .env）

### 10.3 首次配置向导（软引导）

首次启动检测到 KEY 未就绪时弹出配置向导：

- **分必需/可选**：
  - 必需：主对话模型（LLM）—— 不配无法教学
  - 可选：视频理解 + 面试语音（DashScope）、资料 RAG 检索（智谱 Embedding）—— 用到才提示
- **.env 回填标记**：检测到 `OPENAI_API_KEY` 等，自动标记"已就绪 · 来自 .env"，不强迫重填
- **不阻塞**：主模型就绪即显示"✅ 核心教学功能可用"，可点"开始学习"进入；缺的可选 KEY 在用户触发对应功能时再弹"该功能需要 XX Key"
- **获取链接**：每个 Key 旁带"获取 →"按钮，一键跳转各厂商控制台

### 10.4 settings 页面增强

按功能分组：

- **对话模型（LLM）**：沿用现有 UI（`settings.tsx`），支持多 provider + 设默认 + 测试连接。回填的配置带"来自 .env"标记。
- **增强能力**（新增分组）：
  - 🎬 视频理解 + 🎤 面试语音 → DashScope（CosyVoice + Paraformer）单一 Key 覆盖
  - 📚 资料 RAG 检索 → 智谱 GLM Embedding
- **优先级说明卡片**：底部提示"环境变量配置的 Key 会自动回填并标记；在 UI 修改后，以 UI 配置为准（DB 覆盖 .env）"

### 10.5 技术要点

- **DB 扩展**：现有 `LlmConfig`（LLM 配置）模型扩展，或新增 `ServiceConfig` 模型承载 Embedding/视频 KEY（具体 schema 在实现计划细化）
- **回填逻辑**：server 启动时（或首次访问 settings 时）读取 `.env`，对 DB 中缺失的配置执行一次性回填，幂等
- **.env.example 同步**：`.env.example` 注释更新为"这些变量会在首次启动时自动回填到 UI，之后可在模型设置页管理"，避免用户困惑两套体系关系
- **加密**：新增 KEY 沿用现有 `LLM_ENCRYPTION_KEY` 加密机制入库

> ⚠️ Phase 0 是后续所有 AI 功能的可用性前提。语音迭代（2.1.0）的 DashScope Key 依赖 Phase 0 纳入 DB，否则语音功能没 Key 又要回头改 .env。
