## ADR-001: 智谱 GLM 作为默认 LLM

**日期**：2026-05-08
**状态**：~~已采纳~~ → 已废弃（见多模型支持，默认已切换为 DeepSeek `deepseek-v4-flash`）

### 决策

使用智谱 GLM（OpenAI 兼容接口）作为默认 LLM，模型 `glm-4-flash`。

### 背景

初期规划使用 OpenAI GPT-4o / Claude，实际开发时因网络和成本考虑改用智谱。

### 理由

- 国内直连，无需代理
- OpenAI 兼容接口，可直接使用 AI SDK 的 `createOpenAI` provider
- `glm-4-flash` 性价比高，适合高频教学对话
- 可用模型：`glm-4-flash`（默认）、`glm-4.5`、`glm-5`

### 影响

- `OPENAI_BASE_URL` 指向智谱 API
- 所有 Agent（Tutor、Diagnostic、Roadmap）均使用同一 provider
- 可通过修改 `.env` 切换到其他 OpenAI 兼容接口

**日期**：2026-05-08

**日期**：2026-05-08
**状态**：已采纳

### 决策

## ADR-006: Tailwind CSS 4 + CSS 变量设计系统

**日期**：2026-05-08
**状态**：已采纳

### 决策

使用 Tailwind CSS 4 的 `@theme` 定义设计系统，颜色用 CSS 变量而非硬编码。

### 理由

- Tailwind CSS 4 原生支持 CSS 变量，无需 `tailwind.config.js`
- 暖色教育主题（stone + amber）通过 CSS 变量统一管理
- 组件使用语义化变量（`var(--sidebar-bg)` 而非 `bg-slate-900`），方便全局调整

### 约束

- PostCSS 必须用 `module.exports`（不用 `export default`），与 AI SDK 有兼容性问题
- 颜色必须用 `@theme` CSS 变量，不用硬编码色值

## ADR-009: Agent 引擎重构 — BaseAgent + 工具副作用内化

**日期**：2026-05-09
**状态**：已采纳

### 决策

重构 Agent 架构：创建 `BaseAgent` 抽象基类，统一 Provider 创建，将工具副作用从 API 层移入工具 execute 函数。

### 背景

原架构问题：

- 5 个 tool 的 execute 只是 passthrough（返回 `{ success: true }`），副作用散落在 `chat/route.ts` 的 `persistChatTurn` 中（293 行上帝函数）
- 3 个 Agent 各自创建 provider，无共享基础设施
- 无上下文管理，长对话必然崩溃
- 无错误恢复，LLM 调用失败直接 crash

### 理由

- `BaseAgent` 统一 provider 管理、model 获取、重试/fallback 逻辑
- 工具副作用内化后，`chat/route.ts` 从 293 行降至 166 行
- `NodeService` / `MessageService` / `ProfileService` 封装业务逻辑
- `createTutorTools()` 工厂函数使工具可访问 service 层

### 参考

- [Agent引擎优化方案](Agent引擎优化方案.md)
- 参考仓库：Open Agents (Vercel) → ToolLoopAgent + prepareCall/prepareStep
- 参考仓库：Pi Framework → beforeToolCall/afterToolCall 钩子

## ADR-010: 消息截断 + Token 预算估算

**日期**：2026-05-09
**状态**：已采纳

### 决策

实现 `transformMessages()` 消息截断策略和 `estimateTokens()` Token 预算估算。

### 理由

- 智谱 `glm-4-flash` 上下文窗口 128K tokens，但长对话仍需截断
- 保护策略：保护首尾（系统上下文通过 system prompt 传递 + 最近 10 轮），压缩中间
- 单条消息超过 2000 字符时截断
- Token 估算用简单字符计数（中文 1.5 token/字，英文 0.25 token/字）

## ADR-011: LLM 重试 + Fallback 模型链

**日期**：2026-05-09
**状态**：已采纳

### 决策

在 `BaseAgent` 中实现 `executeWithRetry()`（指数退避 1s/2s/4s）和 `executeWithFallback()`（主模型→fallback）。

### 理由

- 智谱 API 偶尔返回 429/500/503，重试可自动恢复
- 指数退避避免加剧限流
- Fallback 模型链提供降级能力（主模型不可用时切换）
- 流式响应不适合重试（流已返回客户端），重试仅用于 `generateObject` 调用（Roadmap/Diagnostic Agent）

## ADR-013: 架构升级 — 三层分离 (Hono + Worker + Next.js UI Only)

**日期**：2026-05-09
**状态**：已采纳

### 决策

将 Next.js 从"全栈框架"降级为"纯前端"，新增 Hono API Server 作为统一 API 入口，Worker 作为独立 Agent 执行进程。

### 背景

当前架构 Next.js API Routes 通过相对路径直接 import Worker 代码（../../../../../worker/src/agent/...），Worker 进程从未运行。Redis/MinIO/BullMQ 配置但未使用。

### 理由

- Hono 轻量、函数式、与 AI SDK 天然兼容（Web Standard API）
- Worker 独立进程支持后台任务、队列削峰、沙箱执行、独立扩缩容
- Next.js 纯前端，消除 UI 和业务逻辑耦合
- 不选 NestJS：AI Teacher 核心是 Agent 不是 CRUD，装饰器体系对 Agent Loop 反而是负担

### 参考

- [架构升级方案](架构升级方案.md)

## ADR-014: Agent 引擎 — 自实现 StateGraph（不采用 LangGraph.js）

**日期**：2026-05-09
**状态**：已采纳

### 决策

借鉴 LangGraph 的核心概念（StateGraph + Checkpoint + 条件边），用 TypeScript + AI SDK 自实现轻量 Agent 引擎，不引入 LangGraph.js。

### 理由

- LangGraph.js 的 TypeScript 是二等公民，功能滞后于 Python 版
- LangGraph 与 AI SDK 不兼容（自有 model abstraction），需要适配层
- LangGraph Checkpointer 耦合 LangChain BaseMessage 格式
- 自实现 ~500 行核心代码可获得 80% 的 LangGraph 价值，避开 90% 的坑
- AI SDK 的 streamText/useChat 保持一等公民地位

### 参考

- [架构升级方案](架构升级方案.md)
- earendil-works/pi: AgentMessage 两阶段转换、compaction
- vercel-labs/open-agents: ToolLoopAgent、prepareCall/prepareStep、Subagent Registry

## ADR-015: 消息持久化 — BullMQ 队列 + Redis Pub/Sub

**日期**：2026-05-09
**状态**：已采纳

### 决策

用户消息先落盘（status: "sending"），投 BullMQ 队列，Worker 消费后执行 Agent，通过 Redis Pub/Sub 推送流式 token，结束后持久化完整消息。

### 背景

当前 `void persistMessages()` fire-and-forget，消息可能丢失。无队列机制，无法排队、重试、崩溃恢复。

### 理由

- 消息先写 DB 保证不丢
- BullMQ 提供重试（指数退避）、并发控制、延迟队列
- Redis Pub/Sub 实时推送流式 token 到前端
- Worker 崩溃后从 Checkpoint 恢复，不从头开始

## ADR-020: BullMQ 队列 + Redis Pub/Sub 实现异步对话架构

**日期**：2026-05-09
**状态**：已采纳

### 决策

采用 BullMQ 消息队列 + Redis Pub/Sub 实现三层分离架构的对话流程：

- Hono API Server 负责 HTTP 接入（POST 发消息 + GET SSE 流订阅）
- Worker 作为独立进程消费 BullMQ 队列执行 Agent
- Redis Pub/Sub 承载流式 token 推送

### 背景

迭代 016 将 Next.js API Routes 迁移到 Hono，但 Agent 仍在 Hono 进程内同步执行。这意味着：

- 单个 Agent 调用阻塞一个 HTTP worker
- 无法水平扩展 Agent 处理能力
- Agent 崩溃会连带影响 API 服务

### 理由

- **BullMQ**：成熟的 Node.js 队列库，基于 Redis，支持优先级、重试、并发控制
- **Redis Pub/Sub**：低延迟推送，无需 WebSocket，SSE 天然兼容
- **解耦**：API 层和 Agent 层独立部署、独立扩展
- **弹性**：Worker 崩溃后自动恢复（BullMQ 重试 + 启动时扫描 orphaned jobs）

### 替代方案

- **直接 HTTP 代理**（Hono → Worker HTTP）：简单但失去队列的持久化和重试能力
- **gRPC**：性能好但引入复杂依赖，与 SSE 不兼容
- **WebSocket**：双向通信但增加前端复杂度，SSE 更简单

### 影响

- 新增依赖：`bullmq`、`ioredis`（Worker + Server）
- Prisma Message 新增 `status` 字段（sending → processing → completed/failed）
- 前端 chat hook 从 AI SDK `useChat` 改为自定义 POST + SSE 实现
- `pnpm dev` 启动三进程：web (:38421) + server (:38422) + worker (:38423)

## ADR-021: Monaco Editor 替换 CodeMirror 6

**日期**：2026-05-13
**状态**：已采纳

### 决策

使用 Monaco Editor（`@monaco-editor/react` + `monaco-editor`）替换 CodeMirror 6 作为代码编辑器。

### 背景

迭代 037 实现了基于 CodeMirror 6 的代码编辑器，但存在几个问题：

- 编辑器和终端共用一个滚动区域，错误信息被挤到视口外看不全
- 缺乏 IntelliSense、代码折叠、搜索替换等 IDE 基础能力
- 不支持多文件编辑（未来 LLM 可能返回多文件工程案例）
- 不支持 HTML/Markdown 预览（未来教学场景需要）

### 理由

- **VSCode 内核**：Monaco 就是 VSCode 的编辑器，IntelliSense / 多光标 / 搜索替换 / 括号着色开箱即用
- **多文件架构**：原生 `monaco.editor.createModel(content, lang, uri)` 支持多文件切换，配合文件树 UI 可做工作区
- **用户心智模型**：目标用户（编程学习者）的心智模型就是 VSCode
- **社区和生态**：`@monaco-editor/react` 封装成熟，Next.js 集成无坑

### 替代方案

- **继续 CodeMirror 6**：轻量（~200KB vs Monaco ~1.5MB gzip），但扩展成多文件/预览需要从零搭建
- **CodeSandbox Sandpack**：专为嵌入式代码编辑设计，但定制化受限、依赖外部服务

### 影响

- 包体积增加（~1.5MB gzip），但编辑器在右侧栏 lazy load，不影响首屏
- 移除 9 个 `@codemirror/*` 依赖，新增 `@monaco-editor/react` + `monaco-editor`
- 同步改造：编辑器/终端可拖拽分割、右边栏可拖拽宽度、终端字体增大
- 架构预留：Output 区 Tab 化（未来加 HTML 预览），Monaco 多 Model（未来加文件树）

## ADR-022: 跨工具 Harness 兼容 — AGENTS.md 单一真理源 + 入口指针

**日期**：2026-06-26
**状态**：已采纳

### 决策

项目开发协议以 `AGENTS.md` 为唯一真理源，详细规则放 `.agents/references/`（工具无关的纯 markdown）。各 AI 编码工具（Claude Code / Cursor / Windsurf / OpenCode / Cline 等）的入口文件（`CLAUDE.md` / `.cursor/rules/*.mdc` / `.windsurfrules` 等）只写一句指针指向 `AGENTS.md`，不重复内容，且全部提交进仓库，使协作者 clone 后零配置即生效。

同时新增 `.agents/references/实施纪律.md`，把单个迭代任务的实施环节固化为 spec → plan → TDD → verify 节律，补强 AGENTS.md §3.0 的执行层。

### 背景

项目原有的 spec 驱动开发模式（迭代计划 + 迭代详情 + references + 文档治理）已是成熟的项目级迭代治理体系，但在"单个迭代任务怎么实施"这一层较薄——AGENTS.md §3.0 只规定"方案 review + 列 todo"，todo 的结构和执行节奏无纪律约束。

考虑过引入 Superpowers（obra/superpowers）的 spec/plan/TDD skills，但 Superpowers 基于 Claude Code 专有的 `SKILL.md` 机制，无法跨工具复用，且其 skills 触发机制（按 description 语义匹配）绑死 Claude Code。

### 理由

- **跨工具兼容**：`AGENTS.md` 是当前跨工具兼容性最好的项目级指令格式，Claude Code / Cline / OpenCode 原生读取，Cursor / Windsurf 通过入口指针引导。纯 markdown 的 references 不依赖任何工具的 skill 机制。
- **单一真理源**：内容只维护一份（AGENTS.md + references），各工具入口只放指针，避免多份副本漂移。
- **零配置协作**：所有入口指针文件提交进仓库，协作者 clone 后任何工具打开即生效。
- **节律而非框架**：吸收 Superpowers 的 plan → TDD → verify → spec-sync 节律，但以文字指令承载，不引入 SKILL.md / skills 文件机制，对单人本地项目不过重。
- **与 1.1.0 一致**：新增的 `实施纪律.md` 是明确可校验的规则（每条 yes/no 判定），未来可被迭代 044（references 自检）的脚本扫描；TDD 强制化挂在迭代 045（Vitest 基础设施）完成之后，顺序对齐。

### 替代方案

- **采用 Superpowers 的 SKILL.md 路线**：放弃。绑死 Claude Code，无法跨工具；skills 触发机制无法在 Cursor/OpenCode 等复用。
- **每个工具维护一份独立规则文件**：放弃。内容重复，漂移成本高，违背单一真理源。
- **不补强实施层，维持现状**：放弃。§3.0 实施层纪律缺失，todo 无结构、无逐项验证、无 TDD 约束，AI 生成代码质量缺乏节律保障。

### 影响

- 新增 `.agents/references/实施纪律.md`
- `AGENTS.md` §3.1 索引表新增"写代码实现迭代任务"行；§3.0 末尾补 todo 结构化 + 逐项验证铁律
- 新增 `.cursor/rules/project.mdc`（alwaysApply 指针）、`.windsurfrules`（指针）

## ADR-023: Vitest 作为单元测试框架

**日期**：2026-06-26
**状态**：已采纳

### 决策

采用 Vitest 4 作为单元测试框架，配合 `@vitest/coverage-v8` 做覆盖率。根目录单 `vitest.config.ts` + workspace include 模式，覆盖 `apps/worker` + `apps/server` + `packages/shared`。

### 背景

迭代 045 之前项目零单元测试，仅 50 个 E2E（Playwright）覆盖用户流程。核心逻辑（掌握度门控、上下文压缩、加解密、流式解析）无单元级验证。045 建立单元测试基础设施。

### 理由

- **与 Vite 共享配置**：项目 Web 端已用 Vite 8，Vitest 原生基于 Vite，配置/转译链路一致，无额外开销
- **ESM 原生**：项目 `module: ESNext`，Vitest 一等支持 ESM；Jest 的 ESM 支持需额外配置
- **速度快**：Vite 的依赖预构建 + 增量转译，测试启动快
- **workspace 友好**：`@ai-teacher/*` 通过 pnpm 链接解析到源码（main 指向 src），根目录单配置即可跨 workspace 测试，无需 alias
- **API 兼容 Jest**：`describe`/`it`/`expect` 语法与 Jest 一致，迁移成本低

### 替代方案

- **Jest**：成熟但 ESM 支持需额外配置，不与 Vite 共享配置，启动较慢。放弃。
- **Node.js 原生 test runner**：零依赖但生态薄（无内置 coverage、mock 能力弱），不适合中大型项目。放弃。

### 影响

- 新增 devDependencies：`vitest` ^4.1.9、`@vitest/coverage-v8` ^4.1.9（根 workspace）
- 新增 `vitest.config.ts`（根目录，workspace include 模式）

## ADR-024: 计算性传感器 — ESLint 自定义规则 + 自定义依赖方向检查

**日期**：2026-06-26
**状态**：已采纳（迭代 042）

### 决策

把 AGENTS.md 中的关键规范从「AI 自觉遵守的推断性规则」升级为「工具强制的计算性传感器」：

1. **ESLint 自定义规则 `no-default-export`**（`scripts/eslint-rules/no-default-export.mjs`）：禁止 `export default`，强制命名导出（AGENTS.md §4）。错误信息含 AI 可消费的修复提示（"a positive kind of prompt injection"）。根 `eslint.config.mjs` flat config，按 web/server/worker 分环境。config 文件（eslint/vite/postcss）与 Playwright globalSetup/Teardown 因框架要求 default export 而豁免。
2. **自定义依赖方向检查 `scripts/check-deps.mjs`**（ArchUnit 风格）：强制 AGENTS.md §5 模块依赖方向矩阵（apps/web→shared、apps/server|worker→shared+db、packages 为叶子包）。

### 背景

迭代 042 之前所有质量门禁都是推断性的——依赖 AI 自觉遵守 AGENTS.md 和 references/。Martin Fowler《Harness engineering for coding agent users》提出 computational vs inferential 二分法：computational sensors（linter、结构化测试）确定性高、速度快、成本低，应作为第一道防线。

### 关键偏离：自定义脚本替代 dependency-cruiser

原计划用 dependency-cruiser，但 dependency-cruiser 18 要求 Node `^22||^24||>=26`，与 **CI（Node 20）/ 本地（Node 25）均不兼容**（25 为非 LTS 被排除）。降级版本也无法同时兼容 20 与 25。故改用自定义脚本——纯 Node ESM、无 Node 版本依赖、契合项目既有 `check-doc-consistency.mjs` 计算性传感器范式。原计划「可优化方向」明确允许「自定义脚本或 dependency-cruiser」，故属计划内适配。

### 已知技术债：5 处 server↔worker 依赖方向违规

依赖方向传感器立即发现 6 处真实架构违规（server 与 worker 通过相对路径逃逸互相导入）。其中 `crypto` 已修复（迁移到 `packages/shared/services/crypto.ts`，子路径导出避免 web 打包 node:crypto）。剩余 5 处作为已知技术债以白名单显式豁免，待后续迭代「依赖方向治理」归位：

- `provider-registry`（server 3 处路由 → worker）：server/worker 共享 LLM provider 构造
- `diagnostic`（server diagnostic 路由 → worker agent）：诊断流程 server 同步调用 worker agent
- `profile-service`（server session-detail → worker service）：server/worker 共享 profile 读写

这些违规涉及「server/worker 共享代码归位到哪个包」的架构决策（provider-registry 需 ai SDK、profile-service 需 db、diagnostic 耦合 BaseAgent），非简单挪文件，故单独成迭代处理。白名单豁免是显式且可追踪的（非静默），传感器对新增违规仍强制失败。

### 影响

- 新增 devDependencies：`eslint` ^10.5.0、`typescript-eslint` ^8.62.0（根 workspace）
- 新增 `eslint.config.mjs`（根）、`scripts/eslint-rules/no-default-export.mjs`、`scripts/check-deps.mjs`
- 新增脚本：`pnpm lint`、`pnpm check:deps`（commit 前必过，已加入 CI）

## ADR-025: Prisma 状态字段 Enum 迁移 + Node.status 改名

**日期**：2026-06-26
**状态**：已采纳（迭代 042，吸收迭代 046 Phase 2）

### 决策

将 7 个状态字段从 `String @db.Text` 迁移为 Prisma 原生 Enum，获得编译时类型安全：`SessionStatus`、`TeachingMode`、`NodeStatus`、`MessageRole`、`MessageType`、`MessageStatus`、`SourceType`。同步在 `packages/shared` 补 `TeachingMode`、`SourceType` 两个 Zod enum（其余 5 个已存在）。

### 背景

状态字段用 String 缺少编译时类型安全，拼写错误只能运行时发现。Zod 侧早已用 `z.enum()`，但 Prisma 侧仍是 String，两侧类型不对齐。

### 关键决策：Node.status 值改名

`NodeStatus` 是唯一含连字符的字段（`not-started`/`in-progress`），而 Prisma enum 不支持连字符。两个选项：

- **全部 7 个含改名**（采纳）：`not-started`→`not_started`、`in-progress`→`in_progress`，波及 ~50+ 引用点（server/worker/web/seed/E2E/prompt 片段）。一次性彻底完成，无残留 String 字段，Prisma 与 Zod 类型对齐。
- 6 个无连字符字段 + Node.status 暂留 String（放弃）：留一个不一致的 String 字段，046 Phase 2 仅部分关闭。

选「全部改名」：本地单用户、无外部 API 消费者、seed 幂等，破坏性变更完全可控；Zod enum 与 Prisma enum 值一致是类型安全的必要条件。

### Enum 取值以代码实际为准（非原始方案臆测）

原 046 方案的 Enum 取值与实际代码不符（已修正，见 046 修订记录）：`MessageRole` 实际 `tutor|learner|system`（`user/assistant` 仅 UI 层映射）、`MessageType` 实际 `text|quiz|quiz_response|assessment|system`（`tool-*` 是 SSE 事件非 DB 值）、`SessionStatus` 补 `diagnosing`、`SourceType` 实际 `pdf|markdown`。

### 迁移实施

`prisma migrate dev` 对 String→Enum 无法自动生成 cast（要 drop+recreate 丢数据）且非交互模式不可用，故手写迁移 SQL（`20260626163818_enum_status_fields`）：Node.status 先 `UPDATE` 转换连字符数据，再 `CREATE TYPE` + `ALTER COLUMN TYPE ... USING (...::text)::EnumType`。迁移前核查 DB distinct 值确认全部能装入 enum。迁移后 `migrate status` 显示 up to date 无 drift，50 行 Node 数据成功转换。

### 影响

- `packages/db/prisma/schema.prisma` 新增 7 个 enum，7 个字段 String→Enum
- 新增迁移 `20260626163818_enum_status_fields`（含 Node.status 数据转换）
- `packages/shared` 新增 `TeachingMode`（session.ts）、`SourceType`（source.ts）Zod enum
- Node.status 值全仓改名（`not_started`/`in_progress`），含 seed、E2E、prompt 片段
- `docs/设计/技术架构.md` 数据模型章节已同步实际 schema
- 046 Phase 2（Enum）完成；046 Phase 1（索引）仍待开始

## ADR-026: 中间件角色评估 — Redis 队列/Pub/Sub，MinIO/pgvector 为 009 准备

**日期**：2026-06-26
**状态**：已采纳（迭代 047 Phase 1）

### 决策

明确 Docker Compose 三个中间件（Redis、MinIO、pgvector）的实际角色，消除技术架构文档中"预留"模糊表述，并评估 Redis 缓存层与 MinIO 存储服务的实施时机：

1. **Redis = BullMQ 队列 + Pub/Sub 流式推送骨干**（非缓存，无缓存层）。Redis 不可移除——它是流式对话实时通信的骨干。
2. **MinIO = 为迭代 009 RAG 文件存储准备**，当前未接入应用代码（无 S3 客户端），保留在 docker-compose。
3. **pgvector = 为迭代 009 RAG 向量检索准备**，扩展已安装（pgvector/pgvector:pg16 镜像），当前未接入。
4. **Phase 2（MinIO 存储服务）暂缓**：条件"迭代 009 近期启动"不满足（009 为 P3、1.2.0、待开始），实施会产生半成品代码。
5. **Phase 3（Redis 缓存）不实施**：本地单用户场景 DB 查询压力极低，引入缓存层增加 DB↔Redis 双写一致性复杂度，ROI 为负。

### 背景

迭代 047 之前，技术架构.md 对中间件角色的描述自相矛盾且过时：

- 第 21 行「Redis | 会话状态 + 对话缓存（预留）」**错误**——Redis 实际用于 BullMQ 队列 + Pub/Sub 流式推送，从未用于会话状态/对话缓存。
- 第 566 行端口表「Redis | 缓存」**错误**。
- 第 704 行「BullMQ + Redis Pub/Sub | 可靠持久化 + 实时流式推送」**正确**——同文档自相矛盾。

实际 Redis 数据流：worker `run-agent-loop.ts` 通过 `publisher.publish()` 把流式 token/SSE 事件发到 Redis channel `chat:${sessionId}`；server `chat.ts` 用 `subscriber` 订阅后通过 SSE 推前端。这是流式对话的实时通信骨干。

### 关键评估：Redis 缓存层 ROI（Phase 3，不实施）

**可能收益**：热门会话最近 N 条消息缓存、会话状态列表查询走 Redis。
**不值得（本地单用户定位）**：

- 当前单用户本地使用场景，DB 查询压力极低，缓存无性能瓶颈可解
- 引入缓存层增加 DB↔Redis 双写一致性复杂度（cache invalidation 策略）
- BullMQ 已依赖 Redis，但那是为了队列可靠性 + Pub/Sub 实时通信，不是为了性能缓存

故不在技术架构.md 标注模糊的"预留"，而是明确「Redis = 队列 + Pub/Sub，无缓存层」。

### 关键评估：MinIO 存储服务时机（Phase 2，暂缓）

迭代 009（学习资料上传 + RAG）依赖 MinIO 存储原始文件 + pgvector 向量化。但 009 为 P3、1.2.0、待开始，不在近期。若现在实施 MinIO 存储服务（`@aws-sdk/client-s3` + storage.ts + upload 路由），会产生无人调用的"半成品"代码（047 风险章节明确警告）。故暂缓，等 009 启动时与 RAG 链路一并实施。

MinIO/pgvector 保留在 docker-compose：本地 `docker up` 全启无额外成本（空闲 MinIO 占用极低），009 最终要用，移除后重新加回反而折腾（docker-compose + .env.example + check:docs 同步）。符合「补强优先于替换」——不破坏既有结构，只补强文档清晰度。

### 影响

- `docs/设计/技术架构.md` 修正 4 处：技术栈表（DB/队列/存储三行）、端口表（Redis/MinIO）、Docker Compose 章节（加中间件角色注释）、环境变量章节（MINIO\_\* 加注释）
- 无代码变更、无 docker-compose 变更、无 .env.example 变更
- 047 Phase 2/3 暂缓，条件记录于本 ADR + 047 详情 + 迭代计划

## ADR-027: 依赖方向治理 — server/worker 共享代码归位到 packages/shared/services

**日期**：2026-06-26
**状态**：已采纳（迭代 048，1.1.0 收官）

### 决策

迭代 042 依赖方向传感器发现的 5 处 server↔worker 路径逃逸违规，全部归位到 `packages/shared/src/services/`（子路径导出 `@ai-teacher/shared/services/*`，不进 barrel）。三种归位策略：

1. **provider-registry（3 处）**：自包含模块（仅依赖 ai SDK），整体迁移到 `shared/services/provider-registry.ts`。worker 内部所有引用（chat-turn/compaction/base-agent/delegate-task/generate-roadmap）改用 shared 子路径，删除 deprecated `provider.ts` 包装 + 原 `provider-registry.ts`。
2. **profile-service（1 处）**：纯 CRUD 领域服务（仅依赖 `@ai-teacher/db` prisma），迁移到 `shared/services/profile-service.ts`。**shared 新增 `@ai-teacher/db` 依赖**，从叶子包变为可依赖 db 的共享业务层。
3. **diagnostic（1 处，3a 重构提取）**：原 `extends BaseAgent` 深度耦合。把 `BaseAgent.executeWithRetry` 提炼为 `shared/services/retry.ts` 通用重试工具，diagnostic 改为 `shared/services/diagnostic.ts` 纯函数（retry + generateObject + schema + provider-registry）。**BaseAgent 成为死代码已删除**（仅 diagnostic 两个子类使用，迁走后无引用）。

### 背景

迭代 042 建立依赖方向传感器后，5 处 server 通过 `../../../worker/src/...` 相对路径逃逸导入 worker 源码的违规被白名单豁免（ADR-024）。本迭代逐项归位，`check-deps.mjs` 的 `KNOWN_VIOLATIONS` 白名单清空，依赖方向矩阵零违规零豁免。

### 关键决策：shared 子路径导出 + 不进 barrel（web 打包隔离）

provider-registry/diagnostic 依赖 ai SDK（~2MB），若进 shared barrel（`index.ts`），web 经 barrel 导入会被迫打包 ai SDK，破坏首屏。解法：复用 042 为 crypto 建立的 `./services/*` 子路径导出模式——`packages/shared/package.json` 的 `exports` 字段：

```json
"exports": {
  ".": "./src/index.ts",
  "./services/*": "./src/services/*.ts"
}
```

- barrel `index.ts` 只 re-export schemas（纯类型/zod，web 安全），**不 re-export services**
- services 仅通过 `@ai-teacher/shared/services/*` 显式子路径导入
- web tsconfig 无 `@ai-teacher/shared/*` 通配 + web 源码无 services 子路径导入 → web 不会打包 ai SDK
- **验证**：web build 产物最大 chunk 1.42MB，与 042 时一致，ai SDK 未进 web bundle

### 关键决策：shared 打破叶子包约束（shared → db）

profile-service 依赖 prisma。归位选项：迁 shared（破叶子包约束）/ 迁 db（db 掺领域逻辑）/ 新建包。选**迁 shared**：

- 语义正确：profile-service 是 learner-profile 领域服务（含 strengths/weaknesses 合并去重逻辑），非通用数据访问
- 与 provider-registry 策略一致（都用 services/ 子路径），不引入第三种包结构
- db 保持"纯 Prisma 访问叶子包"定位
- 代价：shared 从绝对叶子变为可依赖 db，`check-deps.mjs` 矩阵同步更新 `packages/shared: ["packages/db"]`。这是 shared 成长为共享业务层的位置

### 关键决策：diagnostic 3a 重构（而非 3b 异步化）

diagnostic 深度耦合 BaseAgent，两个选项：

- **3a 重构提取**（采纳）：提炼 `executeWithRetry` 到 shared，diagnostic 改纯函数。保留 server 同步 `await` 语义（用户体验不变），彻底解耦。
- 3b 异步化（放弃）：诊断改走 BullMQ 队列。改了诊断同步语义（同步等 → 异步轮询），且 worker 已有 chatTurn 队列模式，但增加复杂度。

选 3a：保留同步语义、行为不变、解耦彻底。附带收益：BaseAgent 成为死代码被删除（仅 diagnostic 用），worker agent 目录更精简。

### 影响

- 迁移/新建：`packages/shared/src/services/{provider-registry,profile-service,diagnostic,retry}.ts`（crypto 已由 042 建立）
- 删除：`apps/worker/src/agent/{base-agent,provider,provider-registry,diagnostic}.ts`、`apps/worker/src/agent/services/profile-service.ts`
- `packages/shared` 新增依赖：`ai`、`@ai-sdk/openai`、`@ai-sdk/deepseek`、`@ai-sdk/anthropic`（锁 ^6.0.0/^3.0.0 对齐 server/worker）、`@ai-teacher/db`（workspace:\*）
- `packages/shared/package.json` `exports` 子路径模式（042 已建，本次复用）
- `scripts/check-deps.mjs`：矩阵 shared→[db]、`KNOWN_VIOLATIONS` 清空
- `docs/设计/技术架构.md` 项目结构 + 5.3 章节同步
- **1.1.0 全部完成**（042-048），`check:deps` 零违规零豁免
