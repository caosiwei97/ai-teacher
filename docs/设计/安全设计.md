> 版本：v1.0

## 1. 安全边界与威胁模型

AI Teacher 是一个**单用户本地部署**的 AI 辅助教学工具。所有数据（PostgreSQL、Redis、MinIO）存储在本地 Docker 容器中，数据目录 bind-mount 到项目 `./data/`。LLM API 调用仅传输对话内容。

### 威胁模型定位

| 维度 | 定位 |
|------|------|
| 部署模式 | 单用户本地机器 |
| 网络暴露 | localhost only，无公网访问 |
| 数据敏感度 | 用户自己的学习数据 + 用户自己的 API Key |
| 信任边界 | 用户 = 管理员 = 唯一使用者 |

**核心安全关注点**：用户通过 AI 对话生成的代码在 Docker 容器中执行。沙箱逃逸是唯一需要认真对待的威胁。

**不是本项目的威胁**：多租户隔离、用户间数据泄露、DDoS、CSRF、XSS（单用户本地应用，无其他用户需要防护）。

## 2. 代码执行沙箱架构

系统当前存在两条代码执行路径：

### Path A（遗留路径）：Worker → Judge0

```
Worker (agent tool)
  → apps/worker/src/sandbox/client.ts
  → Judge0 POST /submissions（同步轮询模式）
```

- Worker 中的 `executeCode` tool 调用 `submitCode()`
- Judge0 收到 submission 后创建隔离进程执行代码
- Worker 轮询 `/submissions/{token}` 直到执行完成

### Path B（当前路径）：Server → OpenSandbox → execd

```
Server (Hono route)
  → apps/server/src/routes/sandbox.ts
  → apps/server/src/services/sandbox.ts
  → OpenSandbox POST /v1/sandboxes（创建容器）
  → execd POST /code（执行代码，SSE 流式返回）
```

- Server 收到请求后，通过 OpenSandbox API 创建/复用容器
- 容器内运行 `opensandbox/code-interpreter` 镜像，暴露 execd 端口（44772）
- 代码通过 execd `/code` API 提交执行

两条路径共存。Worker 的 `execute-code.ts` tool 当前仍调用 Path A 的 `submitCode`（来自 `sandbox/client.ts`）。Server 路由 `/api/sandbox/execute` 使用 Path B 的 `submitCode`（来自 `services/sandbox.ts`）。

### OpenSandbox Docker 安全配置

配置文件：`opensandbox.toml`

```toml
[docker]
network_mode = "bridge"
host_ip = "host.docker.internal"
drop_capabilities = ["AUDIT_WRITE", "MKNOD", "NET_ADMIN", "NET_RAW", "SYS_ADMIN", "SYS_MODULE", "SYS_PTRACE", "SYS_TIME", "SYS_TTY_CONFIG"]
no_new_privileges = true
pids_limit = 256
```

各项含义：

| 配置 | 值 | 作用 |
|------|-----|------|
| `network_mode` | `"bridge"` | 容器使用独立 bridge 网络，不共享宿主机网络命名空间 |
| `drop_capabilities` | 9 项 | 丢弃 AUDIT_WRITE、MKNOD、NET_ADMIN、NET_RAW、SYS_ADMIN、SYS_MODULE、SYS_PTRACE、SYS_TIME、SYS_TTY_CONFIG |
| `no_new_privileges` | `true` | 禁止子进程通过 setuid/setgid 提升权限 |
| `pids_limit` | 256 | 限制容器内最大进程数，防止 fork bomb |

## 3. 输入防护

### 3.1 危险模式黑名单

Worker 端 `execute-code.ts` 定义了 `DANGEROUS_PATTERNS` 正则数组，在代码提交执行前进行检查：

```typescript
const DANGEROUS_PATTERNS = [
  /import\s+os\b/,
  /import\s+socket\b/,
  /require\s*\(\s*["']fs["']\s*\)/,
  /require\s*\(\s*["']child_process["']\s*\)/,
  /subprocess\./,
  /process\.exit/,
  /__import__/,
];
```

匹配到任一模式时，返回安全检查未通过的错误，不执行代码。

### 3.2 Zod Schema 验证

Server 端 `/api/sandbox/execute` 路由使用 Zod 校验请求体：

```typescript
const executeSchema = z.object({
  source_code: z.string().min(1),
  language_id: z.number().int().positive(),
  stdin: z.string().optional(),
  expected_output: z.string().optional(),
});
```

### 3.3 局限性

| 问题 | 说明 |
|------|------|
| 正则绕过容易 | 字符串拼接（`__imp` + `ort__('os')`）、`eval()`、`exec()`、`getattr()` 等方式可绕过所有正则 |
| 无代码长度上限 | `source_code` 只有 `min(1)`，没有 `max()` 限制 |
| 仅 Worker 端检查 | Server 端路由没有 DANGEROUS_PATTERNS 检查，绕过 Worker 直接调用 `/api/sandbox/execute` 不会触发黑名单 |
| 不覆盖所有语言 | 正则主要针对 Python 和 Node.js，Java/C++ 的危险调用未覆盖 |

**实际影响**：本地单用户部署，这些防护的定位是"防止 AI 生成明显危险的代码"，不是对抗恶意用户。真正的隔离依赖 Docker 容器的 capability drop 和 resource limits。

## 4. 资源限制

### 4.1 代码执行级限制

在代码提交时设置的限制（Path A 和 Path B 均使用相同值）：

| 参数 | 限制值 | 设置位置 |
|------|--------|----------|
| CPU 时间 | 5 秒 | `execute-code.ts` 第 53 行，`sandbox.ts` 路由第 24 行 |
| 内存 | 256 MB（256000 KB） | `execute-code.ts` 第 54 行，`sandbox.ts` 路由第 25 行 |
| 墙钟时间 | 10 秒 | `execute-code.ts` 第 55 行，`sandbox.ts` 路由第 26 行 |

### 4.2 容器级限制

OpenSandbox 创建容器时设置（`sandbox.ts` 第 83-89 行）：

| 参数 | 限制值 | 设置位置 |
|------|--------|----------|
| 容器 CPU | 1 核 | `sandbox.ts` `resourceLimits.cpu: "1"` |
| 容器内存 | 2 GiB | `sandbox.ts` `resourceLimits.memory: "2Gi"` |
| 容器存活时间 | 24 小时（86400 秒） | `sandbox.ts` `timeout: 86400` |
| PID 数量 | 256 | `opensandbox.toml` `pids_limit` |

### 4.3 说明

代码执行级限制（CPU 5s / 内存 256MB）和容器级限制（CPU 1 核 / 内存 2 GiB）是两个不同层次。前者针对单次代码执行，后者针对整个容器进程。容器存活 24 小时是为了复用（避免每次执行都创建新容器），sandbox.ts 中有缓存机制（`cachedSandboxId`）在容器可用时直接复用。

## 5. API 密钥管理

### 5.1 用户自定义密钥的加密存储

文件：`apps/server/src/services/crypto.ts`

- 算法：AES-256-GCM（认证加密）
- 密钥来源：环境变量 `LLM_ENCRYPTION_KEY`（64 字符 hex 字符串，即 32 字节）
- 密文格式：`{iv_base64}:{authTag_base64}:{ciphertext_base64}`
- IV：每次加密随机生成 12 字节
- Auth Tag：16 字节 GCM 认证标签

展示时使用 `maskApiKey()` 函数：保留前 3 位和后 4 位，中间用 `***...` 替代。

### 5.2 沙箱中的 API Key 注入

`sandbox.ts` 的 `buildSandboxEnv()` 函数将 `OPENAI_API_KEY` 和 `OPENAI_BASE_URL` 作为环境变量注入到 OpenSandbox 容器中。

这意味着学生代码可以通过 `os.environ['OPENAI_API_KEY']` 读取到密钥。

代码中有明确的安全注释（`sandbox.ts` 第 55-65 行）：

> Current: single-user local app — injecting API keys is safe (user's own machine, own keys).
>
> Future (web service / multi-user): DO NOT inject raw keys. Implement an API proxy on the server.

**当前评估**：本地部署场景下，用户注入的是自己的 API Key，在自己的机器上执行，这完全可接受。如果未来变为多用户服务，需要改为 server-side API proxy。

## 6. 已知风险与缓解状态

| 风险 | 严重度 | 当前状态 | 缓解建议 |
|------|--------|----------|----------|
| Docker socket 挂载（`/var/run/docker.sock`） | 高 | 本地部署可接受 | OpenSandbox 需要访问 Docker daemon 才能创建沙箱容器。多用户场景需用 Docker-in-Docker 或 rootless Docker 隔离 |
| `OPENSANDBOX_INSECURE_SERVER=YES` | 高 | 本地部署可接受 | 关闭了 OpenSandbox 自身的认证。多用户场景需启用认证 |
| `/api/sandbox/execute` 无认证 | 高 | 本地部署可接受 | Server 所有 API 路由均无 auth middleware，CORS 限制为 `localhost:38421`。多用户场景需加认证 |
| API Key 在沙箱环境变量中暴露 | 中 | 本地用户自己的 Key | 用户代码可读取 `os.environ['OPENAI_API_KEY']`。多用户时改为 server-side proxy |
| 正则黑名单可绕过 | 中 | 防君子不防小人 | 本地部署下依赖 Docker 容器隔离。如需加强，考虑 seccomp profile 或 gVisor |
| 无代码长度上限 | 低 | 暂无实际问题 | `source_code` 缺少 `z.string().max(N)` 限制。可加 truncation 或 size limit |
| 无速率限制 | 低 | 单用户无需 | 多用户时对 `/api/sandbox/execute` 加 rate limiting |
| 沙箱容器 24 小时存活 | 低 | 可接受 | 缩短到 1 小时可减少攻击面，但会增加容器创建频率 |
| MinIO 默认凭据 | 低 | 本地无影响 | `minioadmin/minioadmin`。生产部署时应更改 |

## 7. 本地部署安全建议

### 日常使用

1. **保持 Docker 和容器镜像更新**。定期 `docker pull` 获取最新镜像，尤其是 `opensandbox/code-interpreter` 和 `opensandbox/server`。
2. **不要将端口暴露到公网**。服务端口（38421、38422、38423、25432、26379、29000）绑定在 `localhost`。不要修改为 `0.0.0.0` 或在防火墙中放行这些端口。
3. **使用强加密密钥**。`LLM_ENCRYPTION_KEY` 应使用 `openssl rand -hex 32` 生成，不要使用弱密码。
4. **定期轮换 API Key**。虽然密钥加密存储在本地数据库中，但 API Key 本身应定期更换。

### 环境变量保护

- `.env` 文件已在 `.gitignore` 中，不会被提交到版本控制
- 不要将 `.env` 文件分享给他人或上传到任何云存储
- `OPENAI_API_KEY` 等密钥只在本地 Docker 网络内传输

### 如果要在公网部署

本项目不是为公网部署设计的。如果确实需要远程访问，至少需要：

- 在所有 API 路由上加认证 middleware（JWT / session）
- 启用 HTTPS（反向代理如 Nginx/Caddy）
- 关闭 `OPENSANDBOX_INSECURE_SERVER`
- 用 server-side proxy 替代沙箱内的 API Key 注入
- 用 Docker rootless mode 或 Docker-in-Docker 替代 Docker socket 挂载
- 对所有 API 加速率限制
- 配置 MinIO 使用非默认凭据
