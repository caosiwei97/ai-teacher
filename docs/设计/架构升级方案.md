## 1. 研究背景

本方案基于对两个高质量开源项目的深入研究，提取其架构理念和工程实践，结合 AI Teacher 的实际场景设计。

### 1.1 earendil-works/pi

Pi 是一个三层 monorepo 架构的 AI 编程助手（ai / agent-core / coding-agent），其核心设计值得借鉴：

- **嵌套 while 循环**：外层循环管理 Agent 生命周期，内层循环处理 tool calling。每轮结束后通过 steering message 和 follow-up message 引导下一轮行为，而非简单的 `maxSteps` 黑盒。
- **AgentMessage ≠ LLM Message 双阶段转换**：Agent 内部使用自定义的 `AgentMessage` 结构（包含元数据、工具状态、UI 标记），在发送给 LLM 前才转换为标准 `CoreMessage` 格式。这解耦了内部状态管理和 LLM 通信协议。
- **beforeToolCall / afterToolCall 钩子**：工具执行前后的拦截点，可用于权限检查、日志记录、结果转换、副作用触发。
- **自动压缩与结构化摘要**：当上下文超过 token 预算时，自动调用 LLM 生成结构化摘要（已完成话题、掌握状态、误解点、学习偏好、关键决策），替换原始消息序列。
- **30+ 生命周期扩展事件**：覆盖 Agent 运行全链路的事件系统，支持插件化扩展。

### 1.2 vercel-labs/open-agents

Open Agents 是 Vercel 团队的实验性多 Agent 框架，采用声明式风格：

- **ToolLoopAgent 声明式定义**：Agent 的行为通过声明式配置（system prompt、tools、hooks）定义，而非命令式编排。
- **experimental_context 依赖注入**：通过 `experimental_context` 向工具注入运行时上下文（数据库连接、用户信息、会话状态），工具本身不持有依赖。
- **Subagent Registry + task 工具**：主 Agent 通过 `task` 工具按名称调度子 Agent，子 Agent 以 async generator 方式流式返回结果。`toModelOutput` 控制父 Agent 看到的输出，防止上下文污染。
- **Vercel Workflow 可恢复流式**：基于 Vercel Workflow 实现流式处理的断点恢复，Agent 执行中断后可从 checkpoint 续跑。
- **prepareCall / prepareStep 钩子**：在每次 LLM 调用和每个执行步骤前的拦截点，用于动态修改 system prompt、注入上下文、调整参数。

## 2. 现状诊断

对当前 AI Teacher 代码库的全面审查，按严重程度分级。

### 🔴 严重问题（阻塞扩展）

**Worker 是死壳**

`apps/worker/src/index.ts` 仅 11 行代码，没有任何实际逻辑。所有 Agent 代码（tools、prompt、streaming）被 Next.js API Routes 通过相对路径直接引用：`../../../../../worker/src/agent/...`。Worker 进程从未真正运行，monorepo 中的 worker 包形同虚设。

后果：Agent 逻辑与 Web 服务耦合，无法独立扩展、独立部署、独立测试。

**双写竞争**

`assessMastery` 工具的 `autoAdvance` 逻辑和 `advanceNode` 工具都会写入同一个知识节点的掌握度和状态。两个写入路径没有协调机制，存在竞态条件：

- 用户回答问题 → `assessMastery` 评分 → 触发 `autoAdvance` → 更新节点状态
- 同时，Agent 判断应推进 → 调用 `advanceNode` → 也更新同一节点
- 最后写入的覆盖前者，可能导致掌握度评估结果丢失

### 🟠 中等问题（浪费资源或隐藏风险）

**基础设施未使用**

Redis、MinIO、BullMQ、pgvector 都在 `docker-compose.yml` 中配置并启动，但代码中从未实际使用。每次启动开发环境，这些服务占用内存和端口，却没有发挥作用。

**错误掩盖**

工具函数在数据库写入失败时仍返回 `{ success: true }`。Agent 认为操作成功，继续基于错误前提推理。用户看到"已保存"但实际上什么都没存。

```typescript
// 当前代码的典型模式
try {
  await prisma.node.update(...)
} catch (e) {
  console.error(e)
  return { success: true, message: "已更新" }  // 数据库报错了，但工具说成功
}
```

**Fire-and-forget 持久化**

消息持久化使用 `void persistMessages()` 模式，不等待写入完成。如果数据库响应慢或超时，消息直接丢失，用户刷新页面后看不到之前的对话。

### 🟡 一般问题（影响开发效率和可维护性）

**工具定义重复**

5 个独立工具文件各自定义 schema 和逻辑，`create-tools.ts` 中又有内联定义。同一个工具的参数描述在两处维护，容易不一致。

**页面巨石**

`learn/[sessionId]/page.tsx` 超过 500 行，混合了状态管理、API 调用、消息渲染、输入处理、侧边栏逻辑。修改任何一个功能都要在这个巨大文件中定位。

**无消息队列**

没有速率限制、消息去重、崩溃恢复。用户连续快速发送消息可能触发多个并发 Agent 执行，互相干扰。服务崩溃后，正在处理的消息直接丢失。

**缺少 A2UI 框架**

LLM 输出只有纯文本和 Markdown，无法渲染交互式 UI 组件（评估卡片、代码运行结果、知识图谱可视化、数学公式）。教学场景天然需要丰富的互动界面。

**缺少代码执行沙箱**

编程教学场景中，学生写的代码没有安全执行环境。无法实现"写代码 → 运行 → 看结果 → AI 点评"的教学闭环。

## 3. 架构方案

### 3.1 总体架构：三层分离

当前架构是"Next.js 包揽一切"，API Routes 直接调用 Worker 的 Agent 代码。升级后拆分为三层，各司其职：

> 📎 [**目标系统架构图**（Excalidraw 在线编辑）](https://excalidraw.com/#json=N9_UE3yQg8D2-rW94NDjK,B2rcXkYyDkjpMtTw-9rVJA)

**Client 层（Next.js）**：只负责 UI 渲染和用户交互。不含 API 路由，不直接访问数据库，不运行 Agent 逻辑。通过 HTTP/SSE 与后端通信。

**Server 层（Hono）**：HTTP API 服务。处理认证、请求校验、数据查询、SSE 流式代理。将 Agent 任务推入 BullMQ 队列。

**Worker 层**：真正的计算引擎。从队列消费任务，运行 Agent Engine，执行工具调用，管理上下文压缩，调度子 Agent。

### 3.2 后端框架选型：Hono（不是 NestJS）

选 Hono 不选 NestJS，原因明确：

**AI Teacher 的核心是 Agent 执行，不是 CRUD**。NestJS 的装饰器/DI/模块系统为 CRUD 微服务设计，在 Agent Loop + Tool Calling + Streaming 场景下只增加复杂度，不带来收益。

**Hono 实现 Web Standard API**（Request/Response/Headers），与 AI SDK 天然兼容。NestJS 需要 adapter 桥接，`streamText` 的 `toDataStreamResponse()` 返回标准 Response 对象，在 NestJS 中需要额外转换。

**函数式组合风格更适合 Agent 模式**。Agent 的工具、钩子、中间件天然是函数组合，而非装饰器注入。Hono 的中间件链和 Zod Validator 提供足够的结构化能力。

**更轻量，冷启动更快**。Hono 核心 ~14KB，NestJS 启动需要扫描装饰器、构建 DI 容器、初始化模块树。对于需要快速响应的流式场景，启动时间很重要。

| 维度 | NestJS | Hono | Next.js API Routes |
|------|--------|------|--------------------|
| 复杂度 | 高（装饰器/DI/模块） | 低（函数式中间件） | 中（文件约定） |
| AI SDK 集成 | 需 adapter 桥接 | 原生 Web Standard | 原生兼容 |
| 流式支持 | 需手动管理 Response | 原生 SSE/Streaming | 原生 SSE |
| 冷启动 | 500ms+ | <50ms | N/A（随 Next.js） |
| TypeScript | 装饰器反射 | 原生 TS | 原生 TS |
| 适用场景 | 企业 CRUD 微服务 | API + 流式 + Agent | 全栈小项目 |

### 3.3 保留 Worker 进程

Worker 不是被删除，而是从"死壳"变成真正独立运行的计算进程。

**为什么需要独立 Worker**：

- **后台任务**：RAG 索引构建、学习者画像计算、路径规划生成，这些都是耗时任务，不能阻塞 API 响应。
- **沙箱代码执行**：子进程管理、资源隔离、超时控制需要独立进程。
- **队列消费扩展**：多个 Worker 实例可以并行消费 BullMQ 队列，Agent 计算和 Web 请求量独立扩展。
- **故障隔离**：Agent 执行出错（死循环、内存泄漏）不影响 API 服务可用性。

**通信模式**：

| 场景 | 通信方式 | 说明 |
|------|---------|------|
| 对话流式 | Hono → Worker HTTP → SSE 代理 | Hono 将 SSE 流直接代理给客户端 |
| 后台任务 | BullMQ 队列（Redis） | Hono 入队，Worker 消费 |
| 代码执行 | Worker → Judge0 HTTP | Worker 作为沙箱客户端 |

### 3.4 Agent 引擎设计

借鉴 Pi 和 Open Agents 的设计理念，实现自定义轻量框架。不使用 LangGraph.js。

> 📎 [**Agent StateGraph 工作流图**（Excalidraw 在线编辑）](https://excalidraw.com/#json=VQAgVv7FdMMSvQUwAsRHj,z2jVlERte99s6_AaQHuUOg)

#### 3.4.1 为什么不用 LangGraph.js

- **TypeScript 是二等公民**：LangGraph 以 Python 为主，TS 版本功能滞后、文档不全、社区支持弱。
- **与 AI SDK 不兼容**：LangGraph 有自己的模型抽象层（BaseChatModel），无法直接使用 AI SDK 的 `streamText`、`generateText`。
- **Checkpointer 依赖 LangChain 格式**：LangGraph 的状态持久化绑定 `BaseMessage` 格式，与 AI SDK 的 `CoreMessage` 不通用。
- **流式支持笨拙**：相比 AI SDK `streamText` 的 `toDataStreamResponse()`，LangGraph 的流式方案需要更多胶水代码。
- **包体积**：引入 LangGraph 及其依赖约增加 ~500KB。

#### 3.4.2 为什么自建 StateGraph（借鉴 LangGraph 概念）

核心思路：**借概念，不借代码**。

- 借鉴 LangGraph 的 StateGraph + Checkpoint + conditional edges 概念模型
- 底层使用 AI SDK（`streamText`、`useChat`），不引入 LangChain
- 核心代码约 500 行
- 零外部依赖（除 AI SDK 和 Prisma）

#### 3.4.3 核心组件

**StateGraph：声明式工作流**

```typescript
interface GraphNode<S> {
  name: string
  execute: (state: S, ctx: ExecutionContext) => Promise<S>
}

interface ConditionalEdge<S> {
  from: string
  condition: (state: S) => string  // 返回目标节点名
}

class StateGraph<S> {
  private nodes = new Map<string, GraphNode<S>>()
  private edges = new Map<string, string | ConditionalEdge<S>>()
  private entryPoint: string

  addNode(name: string, execute: GraphNode<S>['execute']): this
  addEdge(from: string, to: string): this
  addConditionalEdge(from: string, condition: ConditionalEdge<S>['condition']): this
  setEntryPoint(name: string): this

  async *stream(initialState: S, ctx: ExecutionContext): AsyncGenerator<StreamEvent> {
    let current = this.entryPoint
    let state = initialState

    while (current !== '__end__') {
      const node = this.nodes.get(current)
      if (!node) throw new Error(`Unknown node: ${current}`)

      // Checkpoint before execution
      await ctx.checkpoint.save(current, state)

      // Execute node
      state = await node.execute(state, ctx)

      // Resolve next node
      const edge = this.edges.get(current)
      if (typeof edge === 'string') {
        current = edge
      } else if (edge) {
        current = edge.condition(state)
      } else {
        current = '__end__'
      }

      yield { type: 'node_complete', node: current, state }
    }
  }
}
```

**Tool Registry 依赖注入**

```typescript
interface ToolExecutionContext {
  prisma: PrismaClient
  sessionId: string
  userId: string
  checkpoint: CheckpointStore
  queue: QueueManager
}

interface ToolDefinition {
  name: string
  description: string
  parameters: z.ZodSchema
  execute: (params: unknown, ctx: ToolExecutionContext) => Promise<ToolResult>
  // 自描述：注入 system prompt 的提示片段
  promptSnippet?: string
  promptGuidelines?: string[]
}

class ToolRegistry {
  private tools = new Map<string, ToolDefinition>()
  private hooks: ToolHooks = {}

  register(tool: ToolDefinition): void {
    this.tools.set(tool.name, tool)
  }

  setHooks(hooks: ToolHooks): void {
    this.hooks = hooks
  }

  async execute(name: string, params: unknown, ctx: ToolExecutionContext): Promise<ToolResult> {
    const tool = this.tools.get(name)
    if (!tool) throw new Error(`Tool not found: ${name}`)

    // beforeToolCall hook
    if (this.hooks.beforeToolCall) {
      const decision = await this.hooks.beforeToolCall(name, params, ctx)
      if (decision.skip) return decision.result
    }

    const result = await tool.execute(params, ctx)

    // afterToolCall hook
    if (this.hooks.afterToolCall) {
      await this.hooks.afterToolCall(name, params, result, ctx)
    }

    return result
  }

  // 收集所有工具的 prompt 片段，注入 system prompt
  getPromptInjections(): string {
    const snippets = Array.from(this.tools.values())
      .filter(t => t.promptSnippet)
      .map(t => t.promptSnippet)
    return snippets.join('\n\n')
  }

  // 转换为 AI SDK tools 格式
  toAISDKTools(ctx: ToolExecutionContext): Record<string, CoreTool> {
    const result: Record<string, CoreTool> = {}
    for (const [name, def] of this.tools) {
      result[name] = tool({
        description: def.description,
        parameters: def.parameters,
        execute: (params) => def.execute(params, ctx),
      })
    }
    return result
  }
}
```

**Checkpoint Store 接口**

```typescript
interface CheckpointStore {
  save(nodeId: string, state: unknown): Promise<string>  // 返回 checkpoint ID
  load(checkpointId: string): Promise<{ nodeId: string; state: unknown }>
  loadLatest(sessionId: string): Promise<{ nodeId: string; state: unknown } | null>
  list(sessionId: string): Promise<CheckpointMetadata[]>
}

// Prisma 实现
class PrismaCheckpointStore implements CheckpointStore {
  constructor(private prisma: PrismaClient) {}

  async save(nodeId: string, state: unknown): Promise<string> {
    const record = await this.prisma.checkpoint.create({
      data: {
        nodeId,
        state: JSON.stringify(state),
        createdAt: new Date(),
      },
    })
    return record.id
  }

  async loadLatest(sessionId: string) {
    const record = await this.prisma.checkpoint.findFirst({
      where: { sessionId },
      orderBy: { createdAt: 'desc' },
    })
    if (!record) return null
    return { nodeId: record.nodeId, state: JSON.parse(record.state) }
  }
}
```

### 3.5 上下文管理

借鉴 Pi 的压缩系统，实现基于 token 预算的动态上下文管理。

**核心策略**：不是按消息条数硬编码截断，而是根据 token 预算动态决策。

```
当前上下文 token 数 > 预算 × 0.8
  ↓
触发压缩
  ↓
调用 LLM 生成结构化摘要
  ↓
替换旧消息，保留最近 N 轮
```

**结构化摘要格式**：

```typescript
interface StructuredSummary {
  completedTopics: string[]          // 已完成的教学话题
  masteryState: Record<string, {     // 各知识点掌握状态
    level: 'none' | 'basic' | 'intermediate' | 'advanced'
    lastAssessed: string
  }>
  misconceptions: string[]           // 发现的误解和错误认知
  learningPreferences: {             // 学习偏好
    preferredExplanationStyle: string
    responseToExamples: string
    pacePreference: 'fast' | 'moderate' | 'slow'
  }
  keyDecisions: string[]             // 对话中的关键决策点
}
```

**增量更新**：每次压缩不是重新生成完整摘要，而是将新内容合并到已有摘要中。已完成话题追加，掌握状态更新，误解列表去重合并。

### 3.6 Tool System

借鉴 Open Agents 的依赖注入模式，重新设计工具系统。

**设计原则**：

- **ToolRegistry + DI**：工具不直接 import 依赖，通过 `ToolExecutionContext` 注入 `prisma`、`sessionId`、`userId`。
- **自描述工具**：每个工具提供 `promptSnippet`（注入 system prompt 的使用说明）和 `promptGuidelines`（使用约束），Agent Engine 自动收集并注入。
- **beforeToolCall / afterToolCall 钩子**：执行前可做权限检查、参数校验、日志记录；执行后可做结果转换、副作用触发、数据持久化。

**新增工具：executeCode**

### 3.7 Multi-Agent（Subagent Registry）

借鉴 Open Agents 的子 Agent 注册和分发模式。

**Registry 架构**：

```typescript
interface SubagentDefinition {
  name: string
  description: string
  systemPrompt: string
  tools: string[]           // ToolRegistry 中的工具名
  maxSteps: number
  toModelOutput: (result: AgentResult) => string  // 控制父 Agent 看到什么
}

class SubagentRegistry {
  private agents = new Map<string, SubagentDefinition>()

  register(def: SubagentDefinition): void {
    this.agents.set(def.name, def)
  }

  get(name: string): SubagentDefinition | undefined {
    return this.agents.get(name)
  }

  // 生成 delegateTask 工具的参数描述
  getAgentDescriptions(): string {
    return Array.from(this.agents.values())
      .map(a => `- ${a.name}: ${a.description}`)
      .join('\n')
  }
}
```

**子 Agent 列表**：

| 子 Agent | 职责 | 工具集 |
|---------|------|-------|
| TutorAgent | 日常教学对话，知识讲解，答疑 | assessMastery, advanceNode, executeCode, getContext |
| DiagnosticAgent | 入学诊断，评估学生当前水平 | assessMastery, getContext |
| RoadmapAgent | 学习路径规划，知识图谱生成 | getContext, updateRoadmap |
| ResearchAgent | 教学资料检索，RAG 增强 | searchKnowledge, getContext |
| AssessmentAgent | 专项测验，阶段性评估 | assessMastery, executeCode, generateQuiz |

**delegateTask 工具**：

父 Agent 通过 `delegateTask` 工具调度子 Agent。子 Agent 以 async generator 流式返回中间结果，完成后 `toModelOutput` 过滤输出，只把关键信息传回父 Agent，防止子 Agent 的详细对话记录污染父 Agent 的上下文。

### 3.8 消息持久化 + 队列机制

> 📎 [**消息流转时序图**（Excalidraw 在线编辑）](https://excalidraw.com/#json=S8ixknxPJpkazBDzeelfP,Tj17RU2UZHgbzVRgK5Unlg)

彻底解决当前 fire-and-forget 模式导致的消息丢失问题。

**完整消息生命周期**：

```
1. 用户发送消息
   → 立即写入数据库（status: "sending"）
   → 返回消息 ID 给前端

2. 入队 BullMQ
   → chat-turn 队列
   → 携带 messageId、sessionId、userId

3. Worker 消费
   → Agent Engine 执行
   → Redis Pub/Sub 实时推送 token
   → Hono 通过 SSE 代理给客户端

4. 流结束
   → 持久化完整 tutor 消息（status: "completed"）
   → 更新用户消息状态为 "completed"

5. 异步后处理
   → 入队 after-chat 队列
   → 更新学习者画像
   → 检查节点推进条件
   → 触发掌握度评估
```

**队列拓扑**：

| 队列名 | 用途 | 并发数 | 重试次数 | 特殊配置 |
|--------|------|--------|---------|---------|
| `chat-turn` | 实时对话处理 | 10 | 3 | 优先级队列，超时 120s |
| `background` | RAG 索引、画像计算、路径规划 | 3 | 5 | 延迟队列，可批量 |
| `after-chat` | 对话后处理（画像、掌握度） | 5 | 3 | 延迟 2s 执行，防止频繁更新 |
| `code-exec` | 沙箱代码执行 | 5 | 2 | 按用户限流，超时 30s |

**消息状态机**：

```
sending → processing → completed
                    → failed（可重试）
                    → timeout（自动重试一次）
```

### 3.9 A2UI（AI-to-UI）方案

选择 Tool-based UI + Component Registry 方案，不采用 Vercel Generative UI（streamUI）。

**为什么不用 Vercel Generative UI（streamUI）**：

- **安全风险**：RSC 执行意味着 LLM 输出直接控制服务端组件渲染，存在注入风险。
- **调试困难**：LLM 生成的 UI 代码出错时，错误堆栈指向 RSC 运行时，定位困难。
- **耦合 Next.js**：绑定 React Server Components，无法迁移到其他前端框架。
- **教学 UI 类型可枚举**：教学场景的 UI 类型是有限的（评估卡片、代码结果、图表、公式、选择题），不需要 LLM 动态生成任意组件。

**UIBlock 方案**：

```typescript
// 判别联合类型
type UIBlock =
  | { type: 'text'; content: string }
  | { type: 'assessment'; question: string; options: string[]; correctIndex: number; explanation: string }
  | { type: 'quiz'; questions: QuizQuestion[] }
  | { type: 'code-result'; language: string; code: string; stdout: string; stderr: string; exitCode: number }
  | { type: 'diagram'; diagramType: 'flowchart' | 'mindmap' | 'sequence'; data: unknown }
  | { type: 'formula'; latex: string; description: string }

// 消息携带 UIBlock 数组
interface ChatMessage {
  id: string
  role: 'user' | 'assistant'
  content: string
  metadata?: {
    uiBlocks?: UIBlock[]
  }
}
```

**前端渲染**：

```typescript
// Component Registry
const UIBlockRenderer: Record<UIBlock['type'], React.FC<{ block: UIBlock }>> = {
  text: TextBlock,
  assessment: AssessmentCard,
  quiz: QuizPanel,
  'code-result': CodeResultBlock,
  diagram: DiagramRenderer,
  formula: FormulaDisplay,
}

// 渲染逻辑
function MessageContent({ message }: { message: ChatMessage }) {
  const blocks = message.metadata?.uiBlocks ?? [{ type: 'text', content: message.content }]
  return blocks.map((block, i) => {
    const Component = UIBlockRenderer[block.type]
    return <Component key={i} block={block} />
  })
}
```

工具直接输出 UIBlock 格式。例如 `executeCode` 工具返回 `code-result` 类型的 UIBlock，前端自动渲染为带语法高亮的代码 + 运行结果面板。

#### 3.13 Prompt 组装改造：Prompt Pipeline 模式

当前 prompt 组装为一次性拼接大字符串（`buildTutorSystemPrompt`），难以维护和 A/B 测试。改为 Pipeline 模式：

```typescript
type PromptLayer = (ctx: AgentContext) => string | Promise<string>;

class PromptPipeline {
  private layers: PromptLayer[] = [];
  pipe(layer: PromptLayer): this { this.layers.push(layer); return this; }
  async build(ctx: AgentContext): Promise<string> {
    const parts: string[] = [];
    for (const layer of this.layers) {
      const result = await layer(ctx);
      if (result) parts.push(result);
    }
    return parts.join("\n\n");
  }
}

// 定义：每个 layer 独立文件，按需组装
const tutorPrompt = new PromptPipeline()
  .pipe(roleDefinition)           // 角色定义（固定层）
  .pipe(coreRules)                // 核心教学规则（固定层）
  .pipe(toolGuidelines)           // 工具使用指引（动态层，来自 ToolRegistry）
  .pipe(learnerContext)           // 学习者画像（动态层，有画像时注入）
  .pipe(roadmapContext)           // 知识图谱（动态层）
  .pipe(currentNodeContext)       // 当前节点详情（动态层）
  .pipe(historySummary)           // 历史摘要（来自 Compaction）
  .pipe(modelOverlay);            // 模型特定覆盖（如 glm-4-flash 特殊指令）
```

**优势**：
- **每个 layer 独立文件**，可单独测试和 A/B 测试
- **动态层**只在有数据时注入（无画像时不注入空段落）
- **Tool 自描述**：`toolGuidelines` 直接来自 `ToolRegistry.toPromptSection()`，新增工具不用改 prompt
- **模型适配**：`modelOverlay` 根据 LLM 模型类型注入不同指令（借鉴 Open Agents 的 model-family overlay）

## 6. 不采纳的方案

| 方案 | 不采纳原因 |
|------|----------|
| **NestJS** | 为 CRUD 微服务设计，装饰器/DI/模块系统在 Agent + Streaming 场景下增加复杂度不带来收益。与 AI SDK 的 Web Standard Response 需要 adapter 桥接。冷启动慢。 |
| **LangGraph.js** | TypeScript 是二等公民，功能和文档落后 Python 版。与 AI SDK 不兼容（自有模型抽象层）。Checkpointer 绑定 LangChain BaseMessage 格式。包体积约 500KB。 |
| **Vercel Generative UI（streamUI）** | LLM 输出直接控制 RSC 渲染存在安全风险。生成的 UI 出错时调试困难。强耦合 Next.js RSC 架构。教学 UI 类型可枚举，不需要动态生成任意组件。 |
| **Monaco Editor** | 包体积 2-4MB，初始化 500ms-1s，单实例内存 30-50MB。教学场景需要在对话气泡中嵌入多个编辑器实例，Monaco 的资源消耗无法承受。移动端支持差。 |
| **MinIO** | MVP 阶段不需要文件存储功能。当前已配置但未使用，先移除减少基础设施复杂度，后续需要时再加回。 |
