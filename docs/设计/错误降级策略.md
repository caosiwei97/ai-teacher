## 1. 错误处理分层架构

系统分三层处理错误，每层职责不同：

```
┌─────────────────────────────────────────────┐
│  Web (Next.js 前端)                          │
│  职责：UI 错误展示、用户提示、请求中断        │
├─────────────────────────────────────────────┤
│  Server (Hono API)                           │
│  职责：请求校验、SSE 错误推送、HTTP 状态码    │
├─────────────────────────────────────────────┤
│  Worker (Agent 引擎)                         │
│  职责：LLM 重试、降级、工具容错、任务状态管理  │
└─────────────────────────────────────────────┘
```

### 各层职责明细

| 层级 | 错误捕获方式 | 降级策略 | 用户感知 |
|------|------------|---------|---------|
| Worker | try/catch 包裹每个任务、LLM 调用重试、工具返回容错 | LLM 重试 → 模型降级 → 硬编码兜底 → 工具静默降级 | 收到错误消息或兜底内容 |
| Server | 路由级 try/catch → 更新 DB 状态 → SSE 错误事件 | 返回 HTTP 错误码 + 推送 SSE error 事件 | 前端展示错误提示 |
| Web | api-client 统一 `!res.ok → throw`、SSE error 事件监听、AbortController | pageError 全页错误 / chatError 自动消失提示 | 错误 UI 或 toast |

**核心原则**：Worker 层尽可能恢复，Server 层透传异常状态，Web 层友好展示。

## 2. 重试与降级策略

### 2.1 LLM 调用重试：executeWithRetry

`BaseAgent` 提供 `executeWithRetry` 方法，对所有 LLM 调用做统一重试：

| 参数 | 值 |
|------|-----|
| 最大重试次数 | 3 |
| 退避策略 | 指数退避：1s → 2s → 4s |
| 可重试错误判定 | 字符串匹配 |

**可重试错误检测逻辑**（通过错误消息字符串匹配）：

```
'429' | '500' | '503' | 'timeout' | 'ECONNRESET' | 'ETIMEDOUT'
```

匹配到以上任一字符串的错误会被重试，其他错误直接抛出。

### 2.2 模型降级：executeWithFallback

`BaseAgent` 定义了 `executeWithFallback` 方法，设计意图是：

```
primary model 调用失败 → 切换到 fallback model 重新调用
```

> **⚠️ 当前状态：该方法已定义但在生产环境中未被 graph pipeline 调用。** 这是一个已知缺陷，参见第 8 节。

### 2.3 结构化降级：generateRoadmap

`generateRoadmap` 工具在 LLM 生成失败时，回退到硬编码的 5 节点模板：

```
LLM 调用失败 → 返回预设的 5 节点学习路线图模板
```

用户仍能获得一份可用的路线图，只是内容为通用模板而非个性化生成。

### 2.4 上下文降级：ContextManager compact

`ContextManager` 在对话历史过长时执行 compact 操作（LLM 压缩摘要）。当 LLM 压缩失败时，降级为截断策略：

```
LLM compaction 失败 → 保留最近 20 条消息，丢弃更早的历史
```

这保证了对话能继续进行，代价是丢失较早的上下文。

## 3. 超时控制

系统中存在以下超时机制：

| 位置 | 超时值 | 实现机制 | 说明 |
|------|--------|---------|------|
| chat-turn processor | 120s | `Promise.race` | 单轮对话处理总超时，超时后消息标记为 failed |
| sandbox 代码执行 | 5s CPU / 10s wall | 沙箱内置限制 | 用户代码的 CPU 时间和墙钟时间上限 |
| sandbox 容器启动 | 60s | 轮询循环 | 等待沙箱容器就绪 |
| sandbox 结果轮询 | 15s（30 次 × 500ms） | 轮询循环 | 获取代码执行结果的最大等待时间 |

**超时后的行为**：

- chat-turn：消息状态更新为 `failed`，通过 SSE 推送错误事件，前端展示错误提示
- sandbox：返回 `{success: false, error: "执行超时"}`，Agent 向用户说明超时原因
- sandbox 轮询：向上层抛出超时错误，由调用方的 try/catch 处理

## 4. BullMQ 任务错误处理

### 4.1 孤儿任务恢复

Worker 启动时，主动查询状态为 `processing` 的消息，将其标记为 `failed`：

```
Worker 启动 → 查询 status=processing 的消息 → 批量更新为 failed → 发布 SSE 错误事件
```

这解决了 Worker 异常重启后任务卡在 processing 状态的问题。

### 4.2 单任务错误处理

每个 BullMQ 任务的处理流程：

```
任务开始
  ↓
try {
  执行 Agent 处理逻辑
} catch (error) {
  更新消息状态为 'failed'
  发布 SSE error 事件
}
  ↓
任务完成（无论成功或失败）
```

### 4.3 优雅关闭

Worker 监听 `SIGTERM` / `SIGINT` 信号：

```
收到信号 → 停止接收新任务 → 等待当前任务完成 → 关闭 Worker
```

### 4.4 已知缺陷

> **无死信队列（Dead Letter Queue）**：失败的任务不会被自动重试或归档到单独队列，仅标记为 failed 后不再处理。参见第 8 节。

## 5. LLM 特定错误处理

### 5.1 速率限制检测

系统通过错误消息字符串匹配检测 429 速率限制：

```
error.message.includes('429') → 判定为速率限制 → 纳入重试（指数退避）
```

这种方式依赖 LLM Provider 返回的错误消息格式，不够健壮。

### 5.2 Token 溢出预防

`ContextManager` 管理 token 预算：

| 参数 | 值 | 说明 |
|------|-----|------|
| `tokenBudget` | 6000 | 单次 LLM 调用的 token 预算上限 |
| `COMPACT_THRESHOLD` | 0.8 | 当 token 使用量达到预算的 80% 时触发 compact |

```
对话历史 token 估算
  ↓
使用量 ≥ 80% 预算？
  → 是：执行 compact（LLM 压缩）
    → compact 失败？→ 截断（保留最近 20 条）
  → 否：继续正常处理
```

### 5.3 Provider 路由

`provider-registry.ts` 管理 LLM Provider 配置：

- 支持 8 个 Provider + `MOCK_LLM` 测试模式
- 配置获取链路：`DB 配置 → 环境变量 Provider`

```
获取 LLM 配置
  ↓
数据库中有对应配置？
  → 是：使用数据库配置的 provider + model
  → 否：回退到环境变量中的默认 provider
```

## 6. 前端错误处理

### 6.1 SSE 流错误处理（use-chat-stream）

| 错误场景 | 处理方式 |
|---------|---------|
| SSE `error` 事件 | 解析错误消息，设置 `chatError` 状态 |
| 网络中断 | AbortController 取消请求 |
| `AbortError` | 静默抑制（用户主动取消不弹错误） |
| 非 AbortError | 展示错误提示 |

### 6.2 页面级错误状态

前端维护两个层级的错误状态：

| 状态 | 展示方式 | 触发场景 |
|------|---------|---------|
| `pageError` | 全页错误 UI（需用户操作恢复） | 页面加载失败、会话不存在等致命错误 |
| `chatError` | 自动消失提示（5 秒后隐藏） | 单条消息发送失败、流中断等非致命错误 |

### 6.3 API 调用错误处理（api-client.ts）

所有 API 调用遵循统一模式：

```typescript
// 一致的错误处理模式
if (!res.ok) {
  throw new Error(`请求失败: ${res.status}`)
}
```

所有 `fetch` 调用在 `!res.ok` 时抛出错误，由调用方的 try/catch 统一捕获。

### 6.4 已知缺陷

> **无 React ErrorBoundary**：项目中没有 `error.tsx` 文件，未使用 Next.js 的错误边界机制。组件级渲染错误会导致整页崩溃，无法优雅降级。参见第 8 节。

## 7. 工具级错误模式

### 7.1 数据库工具（静默降级）

`recordStrength` 和 `recordMisconception` 等数据库写入工具采用静默降级模式：

```
try {
  写入数据库
} catch (error) {
  console.error('记录失败', error)
  return { success: true }  // 仍然返回成功
}
```

**设计意图**：学习记录的写入失败不应中断对话。即使数据未能持久化，教学对话仍能继续。

**副作用**：用户和系统都不会感知到数据丢失，可能导致学习进度记录不完整。

### 7.2 代码执行工具（executeCode）

```
安全预检查（禁止危险代码模式）
  ↓
通过 → 调用 sandbox 执行
  ↓
sandbox 返回结果
  ↓
成功 → {success: true, output}
失败 → {success: false, error}
```

安全预检查不通过时直接返回错误，不调用 sandbox。

### 7.3 任务委派工具（delegateTask）

```
try {
  执行委派任务
} catch (error) {
  return { success: false, error: String(error) }
}
```

通用 catch-all 模式，确保工具总是返回结构化结果，不会因异常导致 Agent 执行中断。

## 8. 已知缺陷与改进方向

| 编号 | 缺陷 | 当前状态 | 改进建议 |
|------|------|---------|---------|
| 1 | `executeWithFallback` 已定义但未在 graph pipeline 中调用 | 模型降级功能形同虚设 | 在 graph 节点执行入口集成 fallback 调用 |
| 2 | 无 Next.js ErrorBoundary（无 `error.tsx` 文件） | 组件渲染错误导致整页白屏 | 为关键路由添加 `error.tsx` 错误边界 |
| 3 | 无用户侧重试 UI | 错误发生后用户只能刷新页面 | 添加"重试"按钮，支持消息级别的重新发送 |
| 4 | 外部服务无熔断器（Circuit Breaker） | LLM Provider 持续故障时无保护 | 引入熔断模式，连续失败后自动切换 Provider |
| 5 | BullMQ 无死信队列 | 失败任务仅标记 failed，不重试不归档 | 添加 DLQ，支持自动重试和人工干预 |
| 6 | 重试判定依赖字符串匹配 | 对错误消息格式变化敏感 | 引入结构化错误类型（自定义 Error 类） |
| 7 | Server 端 suggest-reply 等直接 LLM 调用无重试 | 一次性失败即终止 | 对关键 Server 端 LLM 调用增加重试逻辑 |
| 8 | 数据库工具静默吞掉错误 | 学习记录丢失无感知 | 添加本地缓存或队列，延迟重试写入 |
