# 迭代 048：依赖方向治理 — server/worker 共享代码归位

> 优先级：P2 | 分类：优化 | 状态：⬜ 待开始
> E2E：无新增（内部架构重构）
> 来源：迭代 042 依赖方向传感器发现的 5 处已知技术债（见 `决策记录.md` ADR-024）

## 背景

迭代 042 建立依赖方向计算性传感器（`scripts/check-deps.mjs`）后，立即发现 6 处 server↔worker 通过相对路径逃逸互相导入源码的违规。其中 `crypto` 已在 042 修复（迁移到 `packages/shared/services/`，子路径导出）。剩余 5 处因涉及「共享代码归位到哪个包」的架构决策，作为已知技术债白名单豁免，待本迭代解决。

## 待解决的 5 处违规

| #   | 违规                                               | 涉及模块                                            | 耦合度                                    |
| --- | -------------------------------------------------- | --------------------------------------------------- | ----------------------------------------- |
| 1-3 | server 3 处路由 → worker `provider-registry`       | `apps/worker/src/agent/provider-registry.ts`        | 自包含（仅依赖 ai SDK + ai/test），可移动 |
| 4   | server diagnostic 路由 → worker `agent/diagnostic` | `apps/worker/src/agent/diagnostic.ts`               | 深度耦合 worker `BaseAgent`               |
| 5   | server session-detail → worker `profile-service`   | `apps/worker/src/agent/services/profile-service.ts` | 依赖 `@ai-teacher/db`(prisma)             |

## 目标

1. 为 server/worker 共享代码确定正确的归位包，消除 5 处路径逃逸
2. 从 `check-deps.mjs` 的 `KNOWN_VIOLATIONS` 白名单逐项移除
3. `pnpm check:deps` 零豁免通过

## 可优化方向

### 1. provider-registry（3 处，最高优先）

server 和 worker 都需要 LLM provider 构造（server 用于同步快问/建议回复/配置校验，worker 用于队列消费）。`provider-registry.ts` 自包含（仅依赖 `@ai-sdk/*` + `ai` + `ai/test`），是干净的共享候选。

**难点**：`packages/shared` 当前 barrel 被 web 导入，若 provider-registry 进 barrel 会让 web 打包 ai SDK。需用子路径导出（`@ai-teacher/shared/llm`，类似 042 已建立的 `@ai-teacher/shared/services/*` 模式）或新建 `packages/llm` 包。`ai/test` 的 MockLanguageModelV3 是 dev-only，需拆分。

### 2. profile-service（1 处）

`profile-service.ts` 操作 `LearnerProfile`（依赖 prisma）。`packages/shared` 是叶子包不能依赖 db。可归位到 `packages/db`（数据访问层）或新建共享服务包。

### 3. diagnostic（1 处，最难）

`diagnostic.ts` 耦合 worker 的 `BaseAgent`。server 同步调用诊断 agent 是架构选择——是否应改为通过 BullMQ 队列异步化（server 入队、worker 执行）？若是，server 不再需要直接导入 diagnostic，违规自然消除。需先决策诊断流程的同步/异步归属。

## 实施步骤

1. **决策**：确定各模块归位（共享包子路径 / 新建包 / 异步化），记录 ADR
2. **provider-registry**：迁移 + 拆分 mock，更新 3 处 server 路由导入，移除白名单 3 项
3. **profile-service**：迁移，更新 session-detail 导入，移除白名单 1 项
4. **diagnostic**：决策同步/异步后实施，移除白名单 1 项
5. `pnpm check:deps` 零豁免通过 + 全量回归

## 验收标准

- [ ] `pnpm check:deps` 零豁免通过
- [ ] `pnpm typecheck` + `pnpm test:unit` + `pnpm --filter @ai-teacher/web build` 通过

## 风险

- diagnostic 的同步/异步改造可能影响诊断流程的响应时延与用户体验
- provider-registry 迁移需小心 web 不打包 ai SDK（子路径隔离）
- 共享包定位决策影响后续 1.2.0 RAG 迭代的代码组织

## E2E 影响

无新增（纯内部重构，行为不变）；建议跑分类 A（诊断）回归确认 diagnostic 改造无行为变化
