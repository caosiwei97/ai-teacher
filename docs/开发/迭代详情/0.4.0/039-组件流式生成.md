# 迭代 039 — 组件流式生成

> 状态：⬜ 待开始 | 日期：—

## 目标

让 renderUI 产出的卡片/交互组件像文字一样流式生成——用户不需要等待所有组件生成完毕，每解析出一个 block 就立即渲染。

## 背景

### 当前行为

```
LLM 生成 renderUI({blocks: [table, callout, comparison, ...]})
                    ↓ 等整个 JSON 参数生成完毕
       tool-call 事件 → execute() → ui-blocks SSE 事件（整个数组）
                    ↓
       前端一次性渲染所有组件（原子出现）
```

- 文字通过 `text-delta` 逐字流式
- 组件（renderUI blocks）要等 5-10 秒全部生成后才"啪"一下出现
- 体感割裂：文字在流，组件在等

### 期望行为

```
LLM 增量生成 tool-input-delta JSON 片段
       ↓ Worker BlockParser 实时解析
       block #1 完成 → SSE ui-block-delta → 前端立即渲染
       block #2 完成 → SSE ui-block-delta → 前端立即渲染
       block #3 完成 → SSE ui-block-delta → 前端立即渲染
```

### 技术基础

AI SDK v6 的 `fullStream` 已原生支持增量工具参数流式：

| 事件 | 含义 | 当前代码 |
|------|------|---------|
| `tool-input-start` | 工具调用开始，含 toolName | ❌ 忽略 |
| `tool-input-delta` | 增量 JSON 片段 (`delta: string`) | ❌ 忽略 |
| `tool-input-end` | 参数流式结束 | ❌ 忽略 |
| `tool-call` | 完整解析后的参数 | ✅ 当前唯一使用 |

DeepSeek provider 使用 `StreamingToolCallTracker`，完全支持上述事件。

## 方案

### SSE 协议扩展

新增 2 个事件类型（保留原有 `ui-blocks` 作为 fallback）：

```typescript
// 1. 组件开始流式生成（前端显示 skeleton）
{ type: "ui-stream-start", data: { toolCallId: string } }

// 2. 增量单个 block（前端立即渲染）
{ type: "ui-block-delta", data: { toolCallId: string, block: UIBlock, index: number } }
```

### 架构变更

```
┌─ Worker ──────────────────────────────────────────┐
│  tutor-graph.ts                                    │
│  for await (event of result.fullStream):           │
│    case "tool-input-start":                        │
│      → 创建 BlockParser 实例                       │
│      → publish("ui-stream-start")                  │
│    case "tool-input-delta":                        │
│      → parser.feed(delta)                          │
│      → for each parsed block:                      │
│          publish("ui-block-delta", {block, index}) │
│    case "tool-input-end":                          │
│      → parser.flush() 处理残余                     │
│    case "tool-call":                               │
│      → 保留原有 tool-call + ui-blocks 事件         │
│        （fallback 兜底）                            │
└───────────────────────────────────────────────────┘
          │ Redis pub/sub
          ▼
┌─ Server ───────────────────────────┐
│  chat.ts: 透明转发（已实现）        │
└────────────────────────────────────┘
          │ SSE
          ▼
┌─ Web ─────────────────────────────────────────────┐
│  use-chat-stream.ts                                │
│  case "ui-stream-start":                           │
│    → 添加 { streamingBlocks: true } annotation     │
│  case "ui-block-delta":                            │
│    → 增量追加 block 到 annotations                 │
│  case "ui-blocks": (原有 fallback)                 │
│    → 原子替换整个 uiBlocks                         │
│                                                    │
│  MessageContent                                    │
│  → blocks 数组逐个 fade-in 渲染                    │
│  → streamingBlocks=true 时显示尾部 skeleton        │
└───────────────────────────────────────────────────┘
```

## Checklist

### Phase 1: BlockParser 核心实现

- [ ] 新建 `apps/worker/src/streaming/block-parser.ts`
  - `StreamingBlockParser` 类：累积 JSON 片段，实时检测完整 block 边界
  - 输入：`feed(delta: string)` 接收增量 JSON 片段
  - 输出：每次检测到完整 block 时回调 `onBlock(block, index)`
  - 策略：追踪 `blocks` 数组的 `[` `]` `{` `}` 嵌套层级，当一个 object 的花括号闭合且后面跟着 `,` 或 `]` 时视为一个完整 block
  - JSON 解析验证：尝试 `JSON.parse` 验证 block 结构合法性，失败则继续累积
  - `flush()` 方法处理最后的残余数据
- [ ] 单元测试验证 BlockParser 在各种 JSON 片段分割场景下的正确性

### Phase 2: Worker 事件扩展

- [ ] 修改 `apps/worker/src/graphs/tutor-graph.ts`
  - 在 `for await (event of result.fullStream)` 循环中新增分支：
    - `tool-input-start`：创建 `StreamingBlockParser` 实例，publish `ui-stream-start`
    - `tool-input-delta`：当 `toolName === "renderUI"` 时调用 `parser.feed(delta)`，每解析出 block 发布 `ui-block-delta`
    - `tool-input-end`：调用 `parser.flush()`
  - 保留原有 `tool-call` → `ui-blocks` 事件路径作为 fallback（不删除）

### Phase 3: Shared 类型

- [ ] 修改 `packages/shared/src/schemas/ui-block.ts`
  - 新增 SSE 事件类型导出（`UIStreamStartEvent`, `UIBlockDeltaEvent`）
  - 新增前端 annotation 类型（`streamingBlocks?: boolean`）

### Phase 4: 前端消费

- [ ] 修改 `apps/web/src/hooks/use-chat-stream.ts`
  - 新增 `ui-stream-start` 事件处理：添加 `{ streamingBlocks: true }` annotation
  - 新增 `ui-block-delta` 事件处理：增量追加 block 到 annotations 的 `uiBlocks` 数组
  - 保留原有 `ui-blocks` 原子替换逻辑（收到 `ui-blocks` 时替换整个数组，兜底）
- [ ] 修改 `apps/web/src/components/ui-blocks/index.tsx`（MessageContent）
  - blocks 增量追加时使用 key 稳定（block index）
  - 新增 streaming skeleton 尾部占位组件
  - 每个 block 使用 CSS `animation: fadeSlideIn` 渐入效果

### Phase 5: 样式与动画

- [ ] 新增 skeleton 组件 `apps/web/src/components/ui-blocks/block-skeleton.tsx`
  - 脉冲动画卡片骨架，视觉上暗示"正在生成"
- [ ] 在 `globals.css` 或组件中定义 `fadeSlideIn` keyframe 动画
  - `opacity: 0 → 1` + `translateY(8px) → 0`，时长 300ms

### Phase 6: 兼容性与回退

- [ ] 确保 `tool-call` 的完整参数作为最终兜底：如果 BlockParser 没有解析出任何 block（JSON 格式异常），`ui-blocks` fallback 仍能正常工作
- [ ] 历史消息加载路径不受影响（从 DB 读取的 annotations 没有 streamingBlocks）
- [ ] `resumeStream` 路径也支持新增事件类型

## 风险

| 风险 | 缓解 |
|------|------|
| JSON 片段拼接解析出错 | BlockParser 严格边界检测 + `tool-call` → `ui-blocks` 完整 fallback |
| LLM 生成不规范 JSON | `tool-call` 事件的完整参数作为最终兜底 |
| 增量渲染布局跳动 | skeleton 占位 + CSS transition |
| DeepSeek 不按顺序生成 block | 实测验证；block index 保证最终一致 |
| 性能：高频 SSE 事件 | BlockParser 节流：每 50ms 最多发一次 `ui-block-delta` |
