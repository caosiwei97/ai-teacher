# 009 — 学习资料上传 + RAG

> 状态：✅ 已完成（Phase 1，2026-06-26） | 分类：🔵 功能 | 优先级：P3 | 依赖：002

## 目标

用户上传资料（PDF / Markdown / URL），Agent 基于资料内容教学。

## 关联迭代

- **047 Phase 2（MinIO 存储）并入本迭代**：storage service 与 RAG 链路一并实施（ADR-026 的暂缓条件"009 近期启动"现满足，不再产半成品）。
- **047 Phase 3（Redis 缓存）不实施**：ADR-026 判定本地单用户 ROI 为负，本迭代不引入缓存层。
- **Phase 2（音视频理解）仍暂缓**：模型选型/成本阈值/短长切分策略待定，本次只做 Phase 1 文本类。

## 架构决策（实施前确定，2026-06-26）

| 决策点             | 选型                                                                  | 理由                                                                                                                                      |
| ------------------ | --------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| Embedding provider | **智谱 GLM embedding-3**，1024 维                                     | 中文教学场景检索质量最优；当前 LLM DeepSeek 无 embedding API；走智谱 OpenAI 兼容端点，复用已装的 `@ai-sdk/openai`，无需新 SDK             |
| 资料归属           | **用户级全局库**（`Source.userId`，不加 sessionId）                   | 复用现有 Source model（无 sessionId 字段，避免冗余 schema 变更）；用户上传一次、跨会话复用；retrieve-context 按 `userId` 过滤（安全边界） |
| 向量存储           | **pgvector**，HNSW 索引，cosine 距离                                  | docker 已用 `pgvector/pgvector:pg16`；Prisma 用 `Unsupported("vector(1024)")` + raw SQL 检索（Prisma 不原生支持 vector 类型）             |
| 异步处理           | **BullMQ 新队列 `source-processing`**                                 | 解析 + embedding 耗时数秒，复用现有队列架构；`Source.status` 状态机驱动 UI 反馈                                                           |
| 检索接入           | **`retrieve-context` 工具**（agent 按需调用）+ system prompt 轻量提示 | agent 主控检索、token 高效；每轮仅在 system prompt 注入"已关联 N 份资料"提示，不自动注入全文（避免每轮消耗 token）                        |

> ⚠️ **向量维度固化**：1024 维一旦写入 schema 与已灌数据，切换 embedding provider 需 drop + 重建 DocumentChunk。选型已定（智谱 embedding-3 1024），后续不轻易变更。

## 数据模型变更

```prisma
// generator 启用 pgvector 扩展支持
generator client {
  provider        = "prisma-client-js"
  previewFeatures = ["postgresqlExtensions"]
}

extensions = [vector]

// Source 新增 status 字段
model Source {
  id        String      @id @default(cuid())
  userId    String
  title     String
  type      SourceType
  content   String?
  fileUrl   String?
  checksum  String?
  status    SourceStatus @default(PENDING)   // 新增
  createdAt DateTime    @default(now())
  @@index([userId])
}

enum SourceStatus {
  PENDING     // 已入库，待处理
  PROCESSING  // 解析/分块/embedding 中
  READY       // 已向量化，可检索
  FAILED      // 处理失败
}

// 新增文档分块 + 向量
model DocumentChunk {
  id         String   @id @default(cuid())
  sourceId   String
  source     Source   @relation(fields: [sourceId], references: [id], onDelete: Cascade)
  chunkIndex Int
  content    String
  tokenCount Int
  embedding  Unsupported("vector(1024)")
  createdAt  DateTime @default(now())
  @@index([sourceId])
}
```

> HNSW 索引（`USING hnsw (embedding vector_cosine_ops)`）由 migration raw SQL 创建，Prisma 不支持 vector 索引 DDL。

## Phase 2 — 视频理解（🔧 进行中 2026-06-26）

> 状态：🔧 进行中 | 来源：用户需求「上传视频 / 粘贴 B站·YouTube 链接 → 分析视频内容进行教学」
> Phase 1 文本类 RAG 链路（chunk→embedding→pgvector→retrieve-context）**零改动复用**；Phase 2 只新增「视频 → 文本」一环。

### 场景与范围

- **核心场景**：上传视频文件 / 粘贴 B站·YouTube 链接 → 视频理解模型分析内容（画面 + 语音）→ 教学向文本 digest → 入库向量化 → Agent 检索教学。
- **砍掉**：纯音频上传（用户反馈「一般不会有」）。`SourceType` 只加 `video`，不加 `audio`。

### 架构决策（实施前确定，2026-06-26）

| 决策点            | 选型                                                                                                                                                                                 | 理由                                                                                                                                                    |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 视频理解 provider | **阿里 DashScope Qwen3.5-Omni**（全模态，原生音视频）                                                                                                                                | 用户选定；同时理解语音讲解 + 画面，最贴合「分析视频内容」；OpenAI 兼容端点，复用 `@ai-sdk/openai` + provider-registry 已有 qianwen base URL，无需新 SDK |
| 调用方式          | OpenAI 兼容 `/compatible-mode/v1/chat/completions`，message content 用 `{"type":"video_url","video_url":{"url":...}}`                                                                | 与现有 provider 抽象一致；模型端抽帧，我方零 ffmpeg                                                                                                     |
| **视频喂模型**    | **DashScope 文件上传 API**：`GET /api/v1/uploads?action=getPolicy` 取凭证 → POST OSS → 拿 `oss://` 临时 URL（48h，~100MB）→ 喂模型（带 `X-DashScope-OssResourceResolve: enable` 头） | spike 实证：base64 内联有体积上限（38MB→50MB base64 触发 413）；DashScope 自家 OSS 内网拉取，任意体积、无需公网暴露                                     |
| 模型可配置        | `VIDEO_MODEL` env（默认 `qwen3.5-omni-plus`）                                                                                                                                        | 长视频可切 `qwen3.6-plus`/`qwen3.7-plus`（旗舰视觉，支持最长 2 小时视频）无需改码                                                                       |
| 平台视频获取      | **yt-dlp**（`youtube-dl-exec`，支持 YouTube/直链；B站 需 SESSDATA cookie）                                                                                                           | 通用稳健；下载到本地 Buffer → 上传 DashScope。B站 反爬 412，靠设置页用户填 SESSDATA 解决（见下）                                                        |
| 文本产出          | 视频理解模型按**教学向 prompt** 产出「逐段讲解要点 + 板书/代码/图表描述 + 关键概念」digest                                                                                           | digest 即 Phase 1 的 `content`，下游 chunk→embed→store 零改动                                                                                           |
| 资料归属          | 复用 Phase 1 用户级全局库 + retrieve-context（type 无关）                                                                                                                            | Agent 接入零改动；system prompt「已上传 N 份资料」已含视频源                                                                                            |

> ✅ **spike 验证（2026-06-27）**：① base64 内联小视频（991KB）成功，但 38MB→50MB base64 触发 `413 RequestTooLarge`（体积上限）；② DashScope 文件上传路径上传 38MB YouTube 18 分钟视频 → Qwen3.5-Omni **准确提取口语讲解内容**（Loop Engineering/MCP/Intent Debt 等真实概念）→ **音视频理解确认**，digest 结构完整。上传路径为生产主路径。
>
> ⚠️ **架构变更（spike 驱动）**：原方案的 base64 内联 + MinIO presigned URL **均弃用**（前者体积受限，后者本地 MinIO 不可被 DashScope 云端访问）。改为 DashScope 文件上传 `oss://` 路径。`getPresignedUrl` 及 `@aws-sdk/s3-request-presigner` 依赖移除。
