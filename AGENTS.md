# AI Teacher — Agent 工作协议

> 本文件是所有 Agent session 的**入口文件**。新 session 启动时必须先读本文件，按协议执行。
>
> **⚠️ 角色分工**：文档更新（开发日志、迭代计划、API文档等）由主 Agent（Sisyphus）负责。子 Agent（task 委托）只负责代码实现，不得修改 `docs/` 目录下的任何文件。

## 1. Session 启动流程（每次必做）

### 1.1 恢复上下文

新 session 打开后，**必须**按顺序读取以下文件：

```
1. AGENTS.md          ← 本文件，协议入口
2. docs/开发/开发日志.md    ← 开发日志，找到上次中断点和未完成项
3. docs/开发/迭代计划.md    ← 迭代计划和当前进度
```

> 需要了解文档全貌时，读取 `docs/README.md`（文档索引）。

### 1.2 确定工作起点

从 `docs/开发/开发日志.md` 最后一条记录中提取：

- **上次中断位置**：最后完成的迭代编号或具体任务
- **未完成事项**：日志中标记为 `⬜` 或 `[ ]` 的待办
- **阻塞问题**：日志中标注的 "阻塞" / "待确认" 项

**如果日志末尾没有明确标记中断原因**，读取 git log 最近 5 条提交推断：

```bash
git log --oneline -10
```

### 1.3 向用户确认

恢复上下文后，向用户汇报：

```
📋 上下文恢复：
- 上次进展：{最后完成的事项}
- 当前迭代：{迭代编号} — {名称}
- 待处理：{未完成列表}

准备继续 {下一项任务}，确认？
```

## 2. 用户意图识别与分类路由

> **核心原则**：每收到用户消息，先识别意图类型，再走对应流程。不同意图有不同的文档同步要求和执行策略。

### 2.0 意图分类表

收到用户消息后，**必须先判断属于以下哪一类**：

| 意图类型 | 识别特征 | 执行策略 | 文档同步 |
|---------|---------|---------|---------|
| **🟢 产品需求** | "加一个…"、"新增…"、"我要…"、"实现…功能" | 按迭代流程执行：讨论方案 → 确认 → 实现 | 需求规格.md（如涉及新功能）+ API接口.md + 技术架构.md（视变更） |
| **🔴 问题修复** | "报错了"、"bug"、"不对"、"这个不行"、"XX 失败了" | 最小化修复，不改无关代码 | 开发日志.md 记录 bug + 修复方案 |
| **🟡 技术改进** | "重构…"、"优化…"、"提升性能…"、"换一个方案" | 必须先讨论方案，用户确认后再执行 | 决策记录.md（新增 ADR）+ 技术架构.md（如涉及架构变更） |
| **🔵 探索调研** | "看看…"、"调研…"、"研究…"、"怎么做的" | 只读，不改代码 | 开发日志.md 记录调研结论 |
| **⚪ 确认/闲聊** | "好的"、"继续"、"对"、"是的" | 不触发开发流程 | 无 |

### 2.0.1 意图识别流程

```
用户消息
  ↓
1. 匹配意图类型（上表）
2. 如果无法确定 → 直接问用户："你希望我做哪一类工作？"
3. 确定意图后 → 走对应执行策略
4. 执行过程中 → 按对应文档同步要求更新文档
```

### 2.0.2 意图 → 文档同步扩展映射

**在 3.3.2 基础映射表之上**，根据意图类型追加同步：

| 意图类型 | 额外必须同步的文档 |
|---------|-----------------|
| **产品需求** | `docs/产品/需求规格.md`（新增/更新用户故事）+ `docs/设计/决策记录.md`（如有技术选型决策） |
| **技术改进** | `docs/设计/决策记录.md`（新增 ADR，说明为什么改、方案对比） |
| **问题修复** | 无额外要求，但必须在开发日志中记录根因分析和修复方案 |

## 3. 开发过程规范

### 3.1 dev-log 记录规则

**时间格式**：`#### YYYY-MM-DD HH:MM — {标题}`（精确到分钟）

**每条记录必须包含**：

```markdown
#### YYYY-MM-DD HH:MM — {标题}

**状态**：✅ 已完成 | 🔧 进行中 | ⏸️ 暂停 | ❌ 失败

**事件**：
- {做了什么}

**代码变更**：
- `path/to/file` — {改了什么}

**验证结果**：
- {测试/构建/手动验证的结果}

**下一步**：{明确的下一步}
```

**记录时机**：
- 每个独立任务完成时（不是整个迭代结束时）
- 发现 bug 时记录问题和修复方案
- 用户做出关键决策时
- session 结束前必须记录当前状态

**按日期归档规则**：

开发日志采用"主文件索引 + 按日期详情文件"结构：

```
docs/开发/开发日志.md              ← 索引文件（里程碑摘要 + 归档目录表 + 继续点）
docs/开发/开发日志/2026-05-08.md   ← 当日详细日志
docs/开发/开发日志/2026-05-09.md   ← 当日详细日志
```

**写入规则**：
- 详细日志（代码变更、验证结果、技术决策）写入 `开发日志/YYYY-MM-DD.md`（当天文件）
- 主文件 `开发日志.md` 只保留：里程碑级摘要 + 归档目录表 + "下次继续点"
- **每个新 session 第一条日志写入前**，检查当天日期文件是否存在，不存在则创建
- 归档目录表格式：

```markdown
| 日期 | 文件 | 主要内容 |
|------|------|----------|
| 2026-05-08 | [2026-05-08.md](开发日志/2026-05-08.md) | 项目初始化 ~ 迭代 010 |
```

### 3.2 迭代计划同步

**每次开始新迭代时**，更新 `docs/开发/迭代计划.md`：

- 将对应迭代的 `[ ]` 改为 `[x]`
- 更新状态列：`⬜ 待开始` → `🔧 进行中` → `✅ 已完成`

### 3.3 文档同步规则（强制执行）

> **核心原则**：文档更新是代码变更的原子步骤，不是事后补充。每个代码变更必须同步完成文档更新，否则该变更不算完成。

#### 3.3.1 代码变更前的文档影响评估

**在开始写代码之前**，必须先回答：本次变更会影响哪些文档？

```
代码变更 → 文档影响评估 → 确认受影响文档列表 → 开始实现 → 同步更新文档 → 验证一致
```

具体做法：在 todo list 中，每个代码任务后面标注受影响的文档。例如：

```
- [ ] 新增 /api/sources 路由 → 影响：API接口.md, 技术架构.md(项目结构)
- [ ] Prisma 新增 Source model 字段 → 影响：技术架构.md(数据模型), API接口.md(响应)
```

#### 3.3.2 文档触发映射表（完整版）

**代码变更时，必须同步更新对应文档**：

| 代码变更 | 必须更新的文档 |
|---------|-------------|
| **新增 API 路由** | `docs/设计/API接口.md` 新增接口文档（方法+路径+请求体+响应+副作用） |
| **修改 API 请求/响应** | `docs/设计/API接口.md` 更新对应接口的请求体/响应格式 |
| **修改 Prisma Schema** | `docs/设计/技术架构.md` 的数据模型章节（复制实际 schema） |
| **新增 Zod Schema** | `docs/设计/API接口.md` 或 `docs/设计/Prompt设计.md` 视归属而定 |
| **新增/修改 Agent 工具** | `docs/设计/Prompt设计.md` 的工具定义（参数、说明） |
| **修改 Agent Prompt** | `docs/设计/Prompt设计.md` 的 prompt 模板 |
| **新增技术决策** | `docs/设计/决策记录.md` 新增 ADR（编号递增） |
| **新增 npm 依赖** | `README.md` 技术栈表格 |
| **修改端口/配置** | `.env.example` + `README.md` 服务端口表 + `docs/设计/技术架构.md` 端口表和环境变量章节 |
| **修改 docker-compose.yml** | `docs/设计/技术架构.md` 的 Docker Compose 章节 |
| **新增页面/路由** | `docs/设计/技术架构.md` 的项目结构章节 |
| **新增共享类型/Schema** | `docs/设计/技术架构.md` 的项目结构章节 |
| **新增 E2E 测试** | 无需更新文档（测试即文档） |
| **修改 UI 组件** | 无需更新文档（代码即文档） |
| **完成迭代** | `docs/开发/迭代计划.md` 状态 `[ ]`→`[x]` + 完成日期 + `docs/开发/开发日志.md` 记录 |
| **修复 bug** | `docs/开发/开发日志.md` 记录 bug 和修复方案 |

> **核心原则**：架构升级方案中的 Excalidraw 图表是文档的一部分，架构变更时必须同步更新图表。

**图表清单**（`docs/设计/架构升级方案.md` 中引用）：

| Agent StateGraph 工作流图 | [在线编辑](https://excalidraw.com/#json=VQAgVv7FdMMSvQUwAsRHj,z2jVlERte99s6_AaQHuUOg) | 3.4 Agent 引擎 |

**触发条件**：以下代码变更必须同步更新对应图表：

| 变更类型 | 需要更新的图表 |

**更新流程**：

```
识别到图表相关变更
  ↓
1. 打开 Excalidraw 在线链接
2. 编辑图表（使用 fontFamily: 2 即 Helvetica 字体，中文标注）
3. 导出新链接（excalidraw_export_to_excalidraw）
4. 更新架构升级方案.md 中的链接
5. 在本表（3.3.3）中同步更新链接
```

**图表风格要求**：
- 字体：`fontFamily: 2`（Helvetica），不用默认手写体
- 语言：中文标注组件名和职责，英文标注技术术语
- 颜色：按层级区分（蓝色=用户层、紫色=API层、绿色=Agent层、青色=数据层、橙色=沙箱层）

#### 3.3.4 文档更新检查清单

**每个独立任务完成时**（不是 session 结束时），必须执行：

```
□ 代码变更已完成
□ 受影响的文档已全部更新
□ 文档中的代码示例/Schema 与实际代码一致（非旧版本）
□ 文档版本号和更新日期已刷新
```

> **如果代码变更已完成但文档未更新，该任务不得标记为 completed。**

#### 3.3.5 文档自检方法

更新文档时，对以下内容做**交叉验证**：

| 文档 | 验证方法 |
|------|---------|
| `API接口.md` | 接口列表 ↔ `apps/web/src/app/api/` 目录下的 route.ts 文件数量一致 |
| `技术架构.md` 数据模型 | Prisma Schema ↔ 文档中的 prisma 代码块完全一致 |
| `技术架构.md` 环境变量 | `.env.example` ↔ 文档中的变量列表完全一致 |
| `技术架构.md` Docker | `docker-compose.yml` ↔ 文档中的 yaml 代码块一致 |
| `Prompt设计.md` 工具 | `apps/worker/src/agent/tools/*.ts` ↔ 文档中的工具定义一致 |
| `迭代计划.md` | 已完成的迭代 ↔ `git log` 中的实际提交 |

### 3.4 质量门控

**提交代码前必须通过**：

```bash
# 1. 构建
pnpm --filter @ai-teacher/web build

# 2. E2E 测试
npx playwright test

# 3. 两者都通过才允许 commit
```

**E2E 测试同步规则（强制）**：

当需求变更涉及 UI 行为、页面路由、API 接口变化时，**必须同步修改对应的 E2E 测试用例**，不能只改代码不改测试。

| 变更类型 | 必须检查/修改的 E2E 文件 |
|---------|----------------------|
| **修改页面路由或重定向行为** | `e2e/home.spec.ts` 及涉及路由的测试 |
| **修改 Chat UI 交互** | `e2e/chat.spec.ts`, `e2e/learn.spec.ts` |
| **修改诊断流程** | `e2e/diagnostic.spec.ts` |
| **修改会话管理（新建/归档/状态）** | `e2e/session-management.spec.ts` |
| **修改掌握度/节点推进逻辑** | `e2e/mastery.spec.ts` |
| **新增 API 路由** | 新增对应 `e2e/*.spec.ts` 或在已有文件中追加用例 |
| **修改推荐话题/首页内容** | `e2e/home.spec.ts` |

**E2E 失败后的处理流程**：

```
E2E 失败
  ↓
1. 分析失败原因：是代码 bug 还是测试用例过时？
  ↓
2a. 代码 bug → 修复代码 → 重新跑 E2E → 通过后提交
2b. 测试过时 → 确认新行为符合预期 → 更新测试 → 重新跑 E2E → 通过后提交
  ↓
3. 绝不允许：跳过测试、删除失败测试、@ts-ignore 测试错误
```

**文档一致性检查**（提交前必须确认）：

```
□ 本次变更涉及的所有文档已更新
□ 文档中的代码片段与实际文件一致
□ 如果新增了文档，docs/README.md 索引已更新
```

**如果测试发现 bug**：
1. 先修复 bug
2. 重新跑测试
3. 全部通过后再提交

## 4. 缓存清除与服务重启（强制）

> **核心原则**：Next.js 缓存和 dev server 状态不会自动感知所有变更，特定操作后必须手动清除缓存并重启。

### 4.1 必须清除 `.next/` 缓存的场景

遇到以下变更时，**必须**在提交前执行：

```bash
rm -rf apps/web/.next
pnpm dev   # 重启 dev server
```

| 触发条件 | 为什么需要清除 |
|---------|-------------|
| 修改 `globals.css` 的 `@theme` 变量或 `@layer` 定义 | Tailwind CSS 4 缓存旧的 CSS 变量值 |
| 修改 `postcss.config.mjs` 或 `tailwind.config.*` | PostCSS 配置变更不触发 HMR |
| 修改 `.env` 或环境变量 | Next.js 在启动时读取环境变量，运行时不会重新读取 |
| 修改 `prisma/schema.prisma` 并执行 migrate | Prisma Client 缓存需要重新生成 |
| 修改 `next.config.*` | Next.js 配置变更需要重启 |
| 页面路由或布局结构发生结构性变化 | Next.js App Router 缓存了路由映射 |

### 4.2 必须重启 dev server 的场景

| 触发条件 | 说明 |
|---------|------|
| 新增/删除 npm 依赖 | 需要重启让新依赖生效 |
| 修改 `tsconfig.json` | TypeScript 编译选项变更 |
| Prisma migrate 后 | 生成的 Client 需要新进程加载 |
| 添加新的 API route 文件（首次） | 部分情况下 Next.js 不会自动检测新文件 |

### 4.3 代码变更后的验证流程

```
代码变更
  ↓
1. 判断是否触发缓存清除（查上表）
  ↓
2a. 不触发 → 依赖 HMR 热更新，直接验证
2b. 触发 → rm -rf apps/web/.next → 重启 dev server
  ↓
3. 构建验证：pnpm --filter @ai-teacher/web build
  ↓
4. E2E 验证：npx playwright test
```

## 5. 大改动讨论机制

> **核心原则**：大改动必须先与用户讨论方案，确认后再执行。避免"改了很多才发现方向不对"的浪费。

### 5.1 什么是"大改动"

满足以下**任一条件**即为大改动，必须先讨论：

| 条件 | 示例 |
|------|------|
| **涉及 3 个以上文件** | 新功能通常跨多个文件 |
| **涉及 Prisma Schema 变更** | 新增/修改 model 或字段（影响迁移） |
| **涉及新增 npm 依赖** | 引入新的库或框架 |
| **涉及 API 接口变更** | 新增/修改/删除 API 路由 |
| **涉及页面路由结构变更** | 新增页面、修改 URL 结构、改变重定向逻辑 |
| **涉及 Agent 工具或 Prompt 变更** | 影响 AI 行为核心逻辑 |
| **涉及架构层面重构** | 如抽象基类、服务层拆分、跨模块影响 |

### 5.2 讨论流程

```
识别为大改动
  ↓
1. 暂停实现，向用户展示方案摘要：
   - 📋 变更范围（涉及哪些文件/模块）
   - 🏗️ 实现方案（怎么改，分几步）
   - ⚠️ 风险和副作用（可能影响什么）
   - 📝 需要同步更新的文档
  ↓
2. 等待用户确认：
   - "确认按此方案执行？" 或
   - 用户可能给出修改意见
  ↓
3. 用户确认后 → 创建 todo list → 开始实现
```

### 5.3 小改动直接执行

以下情况可以**不经讨论直接执行**：

- 单文件 bug 修复（不涉及其他文件）
- UI 文案调整
- CSS 样式微调
- 注释修改
- 配置值微调（端口号、超时时间等）

## 6. 技术约束（不可违反）

- **pnpm workspaces**（不用 npm）
- **命名导出**（不用 default export）
- **中文 UI 文本**
- **注释仅在必要时添加**
- **端口**：Web 38421, Worker 38422, PG 25432, Redis 26379, MinIO 29000/29001
- **LLM**：智谱 OpenAI 兼容接口，默认 `glm-4-flash`
- **PostCSS**：必须用 `module.exports`（不用 `export default`）
- **.env**：通过 symlink `apps/web/.env → ../../.env` 共享
- **Tailwind CSS 4**：颜色用 `@theme` CSS 变量，不用硬编码色值

apps/

packages/

e2e/            — Playwright E2E tests

## 8. Session 结束流程

**session 结束前（或用户说"结束"/"先到这里"）**，必须完成：

1. ✅ 更新 `docs/开发/开发日志.md` — 记录本次 session 所有工作
2. ✅ 更新 `docs/开发/迭代计划.md` — 同步迭代状态
3. ✅ 执行文档自检（3.3.4）— 确认本次变更涉及的所有文档与代码一致
4. ✅ 如有代码变更，确认已提交并推送
5. ✅ 在开发日志末尾标注**下次 session 应从哪里继续**
