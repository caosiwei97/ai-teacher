# AI Teacher — Agent 工作协议

> 本文件是所有 Agent session 的**入口文件**。新 session 启动时必须先读本文件，按协议执行。
>
> **⚠️ 角色分工**：文档更新（开发日志、迭代计划、API文档等）由主 Agent 负责。子 Agent（task 委托）只负责代码实现，不得修改 `docs/` 目录下的任何文件。

---

## 1. Session 启动流程（每次必做）

### 1.1 恢复上下文

新 session 打开后，**必须**按顺序读取以下文件：

```
1. docs/开发/开发日志.md    ← 开发日志，找到上次中断点和未完成项
2. docs/开发/迭代计划.md    ← 迭代计划索引（版本总览 + 当前工作 + E2E 映射）
```

> **按需加载迭代详情**：仅当需要实现某个迭代时，才读取 `docs/开发/迭代详情/{version}/` 下的详情文件。
>
> 需要了解文档全貌时，读取 `docs/README.md`（文档索引）。

### 1.1.1 检查待同步文档变更

读取 `docs/开发/开发日志.md` 时，**首先检查 `## 待同步文档变更` 区域**：

- 如果为空 → 正常继续 §1.2
- 如果有待处理条目 → 加载 `.agents/references/文档同步规则.md` → 按规则消费条目 → 继续 §1.2

### 1.2 确定工作起点

从 `docs/开发/开发日志.md` 的"下次继续"区域提取断点信息。如果不明确，读取 `git log --oneline -10` 推断。

### 1.3 向用户确认

```
📋 上下文恢复：
- 上次进展：{最后完成的事项}
- 当前迭代：{迭代编号} — {名称}
- 待处理：{未完成列表}

准备继续 {下一项任务}，确认？
```

## 2. 用户意图识别与分类路由

收到用户消息后，**必须先判断属于以下哪一类**：

| 意图类型         | 识别特征                        | 执行策略               | 文档同步                           | E2E 测试           |
| ---------------- | ------------------------------- | ---------------------- | ---------------------------------- | ------------------ |
| **🟢 产品需求**  | "加一个…"、"新增…"、"实现…功能" | 讨论方案 → 确认 → 实现 | API接口.md + 技术架构.md（视变更） | 按映射表检查       |
| **🔴 问题修复**  | "报错了"、"bug"、"不对"         | 最小化修复             | 视变更范围                         | 涉及 UI/API 则检查 |
| **🟡 技术改进**  | "重构…"、"优化…"、"换方案"      | 先讨论方案，确认后执行 | 决策记录.md（新增 ADR）            | 涉及行为变更则检查 |
| **🔵 探索调研**  | "看看…"、"调研…"、"怎么做的"    | 只读，不改代码         | 无                                 | —                  |
| **⚪ 确认/闲聊** | "好的"、"继续"、"对"            | 不触发开发流程         | 无                                 | —                  |

**意图识别流程**：匹配意图 → 无法确定则问用户 → 走对应执行策略 → 按文档同步要求更新。

> **E2E 测试映射**：涉及 UI 行为、页面路由、API 接口变化时，必须按 `质量门控.md` 中的映射表同步检查/修改 E2E 测试用例。

## 3. 开发过程规范

### 3.0 方案 Review + Todo 规划前置步骤（不可跳过）

任何涉及代码变更的任务，在动手实现**之前**必须：

1. **Review 迭代方案**（如存在迭代详情文档） — 读取迭代详情 → 对照实际代码状态审查方案的完整性和可行性 → 识别遗漏/风险 → 向用户报告 review 结论。方案 review 通过后才能进入 todo 规划

   **过时性判定**：Review 发现方案已部分/全部实现时：
   - 已全部实现 → 标记迭代为 ✅ 已完成，通知用户，不重复实施
   - 已部分实现 → 列出剩余项，只规划剩余部分
   - 方案与实际实现冲突（代码走了不同路径） → 暂停，向用户报告差异，确认后再行动

2. **读取 `.agents/references/文档同步规则.md`** — 按触发映射表评估本次变更影响哪些文档，在 todo 中标注
3. **读取 `.agents/references/质量门控.md`** — 如变更涉及 UI 行为/页面路由/API 接口，todo 必须包含 E2E 测试检查步骤

> ⚠️ 如果 todo 中没有文档同步和 E2E 检查步骤（且变更触发了映射规则），该 todo 规划视为不合格，必须补充后再开始实现。

> ⚠️ todo 每项必须含「动作 + 涉及文件 + 验证方式」三要素；执行时逐项验证，不跳步、不批量改完再验证。详见 `.agents/references/实施纪律.md`。

### 3.0.1 子 Agent 任务完成后的回收检查

子 Agent（task 委托）只负责代码实现，不修改文档。主 Agent 在子 Agent 完成后，**必须**：

1. 按 `文档同步规则.md` 的触发映射表评估是否有文档需要更新
2. 按 `质量门控.md` 的 E2E 映射表评估是否需要同步修改 E2E 测试
3. 如有，主 Agent 自行完成文档和 E2E 的同步

### 3.1 按需加载详细规则

| 当你要做...                     | 先读                                 |
| ------------------------------- | ------------------------------------ |
| 写代码实现迭代任务              | `.agents/references/实施纪律.md`     |
| 写代码并需要同步文档            | `.agents/references/文档同步规则.md` |
| 提交代码或跑测试                | `.agents/references/质量门控.md`     |
| 涉及 CSS/env/config/prisma 变更 | `.agents/references/缓存与重启.md`   |
| 大范围改动（3+ 文件）           | `.agents/references/大改动讨论.md`   |
| 写开发日志                      | `.agents/references/开发日志规范.md` |

### 3.2 迭代计划同步

每次开始新迭代时：

1. 更新 `docs/开发/迭代计划.md`：状态列 `⬜ 待开始` → `🔧 进行中` → `✅ 已完成` + 完成日期
2. 更新 `docs/开发/迭代详情/{version}/XXX-名称.md`：Checklist `[ ]` → `[x]`

### 3.3 文档时效性规则

Agent 在执行任务过程中，如果发现文档内容与实际代码/项目结构不一致，**必须立即修正文档**，不能只在心里知道。发现不一致 → 立即修正 → 检查其他文档是否引用了相同过时信息。

## 4. 技术约束（不可违反）

- **pnpm workspaces**（不用 npm）
- **命名导出**（不用 default export）
- **中文 UI 文本**
- **注释仅在必要时添加**
- **端口**：Web 38421, Server 38422, Worker 38423, PG 25432, Redis 26379, MinIO 29000/29001, OpenSandbox 2358
- **LLM**：DeepSeek（@ai-sdk/deepseek 原生 provider），默认 `deepseek-v4-flash`
- **PostCSS**：必须用 `module.exports`（不用 `export default`）
- **.env**：通过 symlink `apps/web/.env → ../../.env` 共享
- **Tailwind CSS 4**：颜色用 `@theme` CSS 变量，不用硬编码色值

## 5. 项目结构速查

```
apps/
  web/          — Vite + React 19 (SPA) + Chat UI（纯前端）
  server/       — Hono API Server (REST API + SSE)
  worker/       — Agent Worker (BullMQ 队列消费 + AI SDK streamText + 11 tools)
packages/
  shared/       — Zod schemas, types
  db/           — Prisma schema + seed
docker-compose.yml  — Docker Compose (PG + Redis + MinIO + OpenSandbox)
e2e/            — Playwright E2E tests
scripts/        — 构建/部署/工具脚本
docs/           — 项目文档（中文）
  产品/         — 产品定位
  设计/         — 技术架构、API接口、Prompt设计、决策记录
  开发/         — 迭代计划、开发日志
  测试/         — E2E 测试用例
```

---

## 6. Session 结束流程

session 结束前必须完成：

1. ✅ 更新 `docs/开发/迭代计划.md` — 同步迭代状态
2. ✅ 执行文档自检 — 确认本次变更涉及的所有文档与代码一致
3. ✅ 如有代码变更，确认已提交并推送
4. ✅ 更新 `docs/开发/开发日志.md` 的"下次继续"区域（覆盖写，2-5 行）
